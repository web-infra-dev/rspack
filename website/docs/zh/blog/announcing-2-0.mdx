与此同时，我们也持续投入性能优化。除了不断改进内部算法和数据结构，我们还实现了增量构建、Lazy Barrel、原生文件系统监听器等能力。Rspack 因而逐步发展为一个既兼容 webpack 生态、又具备更高构建性能的打包方案。

这说明 Rspack 已经在越来越多的真实项目中得到验证，并逐步成为社区里可以稳定使用的 bundler 方案。

## 尝试 2.0

### 新用户安装

如果你是第一次使用 Rspack，推荐从一个基于 Rspack 2.0 的新项目开始体验。需要注意的是，Rspack 2.0 要求 **Node.js 20.19+** 或 **22.12+**。

如果你希望获得一套开箱即用、默认由 Rspack 驱动的构建体验，我们推荐直接使用 [Rsbuild](https://rsbuild.rs/) 创建项目：

```bash
npm create rsbuild@latest
```

如果你更希望直接使用 [Rspack CLI](/api/cli)，也可以执行：

```bash
npm create rspack@latest
```

按照提示完成初始化即可。更多安装方式可以参考 [快速上手](/guide/start/quick-start) 文档。

### 老用户升级

Rspack 2.0 带来了不少新特性，同时也移除了历史上已经废弃的一些能力。对于已有项目，建议先阅读 [迁移指南](/guide/migration/rspack_1.x)。文档中已经整理了从 v1 到 v2 的主要不兼容变更，以及对应的迁移方式。

如果你正在使用支持 Skills 的 Coding Agent，我们也推荐借助它来完成迁移。安装 `rspack-v2-upgrade` 之后，Agent 就可以结合你的项目配置和官方迁移规则辅助完成升级流程。对于配置项较多、历史包袱较重的项目，这通常会比手工逐项排查更高效。

```
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## 后续方向

Rspack 2.0 是一个里程碑，也意味着新的开始。

从长期来看，Rspack 团队会将工作重心从“兼容 webpack”逐步转向**构建性能**、**产物体积**、**社区规范**、**Agent 协同**等方向，并结合 Rstack 生态中的一整套开发工具，为开发者提供“AI 友好的一体化 JavaScript 工具链”。

短期内，团队会主要推进以下几项工作：

- 持续改进 tree shaking 和 side effects 分析能力，并探索函数级的模块切分方案，以进一步降低产物体积
- 持续优化构建链路上各个环节的算法和数据结构，并将它们接入到增量构建和持久化缓存当中，以提升各种场景下的构建速度
- 持续增强 ESM 产物的质量和稳定性，并与社区中的其他团队共同探讨 bundler 对 ESM 规范更好的支持方式
- 持续建设 [Rstack Skills](https://github.com/rstackjs/agent-skills)，为开发者提供更多 AI 友好的工具来提升开发效率，降低问题排查、产物分析的成本

## 常见问题

### 2.0 以后，Rspack 是否不再兼容 webpack？

不是。Rspack 仍会保持有限的 webpack 兼容性，并尽可能让现有项目能够继续平滑迁移和使用。

变化在于，从 2.0 开始，Rspack 不再把“兼容”作为唯一目标。当我们在构建性能、产物体积、社区规范、Agent 支持等方向上找到更合适的方案，而这些方案又必须打破一部分历史兼容时，我们会选择做出相应调整。

即便如此，我们仍会尽量提供清晰的过渡方式、升级说明和迁移方案，帮助用户理解这些变化，并以可控的成本完成迁移。

### 2.0 发布后，1.x 版本是否会完全停止维护？

不会。2.0 发布后，Rspack 的主要演进方向会逐步转向 2.x，但这并不意味着 1.x 会立刻完全停止维护。对于仍在使用 1.x 的项目，我们会根据实际情况，在一段较长的时间内继续提供必要的维护，例如关键问题修复或迁移期支持。

但从长期来看，新增能力和主要优化都会优先在 2.x 上推进，因此如果你的项目还停留在 1.x，我们建议尽早规划升级。

<span id="rsc-support"></span>
## 内置 React Server Components 支持

Rspack v2.0.0 及更高版本内置了对 React Server Components 的完整支持。

这套支持主要包括：

1. 框架无关：不依赖任何元框架，提供通用的底层 RSC 构建能力。
2. 编译期检查：在编译阶段拦截违反 RSC 规范的 API 调用，提前规避运行时错误。
3. CSS 支持：在编译阶段收集服务端与客户端组件样式，并在渲染时注入。
4. HMR 支持：同时支持服务端和客户端组件的热更新。

当前有两种使用方式：

- **使用 Rsbuild**：Rsbuild 提供对 React Server Components 的开箱即用支持，可以快速创建一个 RSC 项目。详见 [rsbuild-plugin-rsc](https://github.com/rstackjs/rsbuild-plugin-rsc)。
- **手动配置 Rspack**：你也可以参考当前文档，手动添加 React Server Components 相关配置。

[rspack-rsc-examples](https://github.com/rstackjs/rspack-rsc-examples) 仓库中提供了使用 Rspack 构建 RSC 应用的完整示例。

<span id="portable-cache"></span>
## 可移植缓存 @jerrykingxyz

在 Rspack 1.2 中，我们引入了[持久化缓存](/config/cache#persistent-cache)。不过，这份缓存无法在不同机器、不同路径和不同环境之间复用，因此团队协作、远程构建等场景很难通过缓存来降低构建耗时。

在 Rspack 2.0 中，我们进一步引入了可移植缓存（Portable Cache）。你可以通过 [`cache.portable`](/config/cache#cacheportable) 生成和使用一份可复制的缓存数据：

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

此外，Rspack 2.0 还提供了 [`cache.readonly`](/config/cache#cachereadonly) 选项。开启后，缓存只会从磁盘读取，不会写回。这对于 CI 场景尤其有用，可以直接复用预热缓存而不修改它。

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

<span id="pure-esm"></span>
## 纯 ESM 包 @chenjiahan

为了让 Rspack 的包形态更符合现代 Node.js / ESM 生态，`@rspack/core`、`@rspack/cli`、`@rspack/dev-server` 和 `@rspack/plugin-react-refresh` 等包已正式切换为纯 ESM 形式发布，CommonJS 构建产物也被一并移除。这让模块加载和工具链集成方式更加统一。

因此，你需要将 Node.js 升级到原生支持 `require(esm)` 的 20.19+ 或 22.12+ 版本。

<span id="better-esm"></span>
## 更好的 ESM 产物

对于库构建场景，ESM 产物的质量非常重要。产物越纯净，越有利于静态分析，也越有利于下游继续做代码分割和 tree shaking。

在 Rspack 1.6 中，我们引入了实验性的 [`rspack.experiments.EsmLibraryPlugin`](https://v1.rspack.rs/plugins/rspack/esm-library-plugin)，用一条专用的渲染路径替代此前基于 module concatenation 的 ESM 输出方案。之后的若干个小版本里，这条路径持续被打磨，并逐步接管了产物流水线中更多的环节。到 2.0，它已经完全稳定。

与此同时，我们移除了实验性插件入口。现在你只需要将 [`output.library.type`](/config/output#outputlibrarytype) 配置为 `'modern-module'`，就可以启用这条路径：

```js title="rspack.config.mjs"
export default {
  output: {
    library: {
      type: 'modern-module',
    },
  },
};
```

与通用的 [`output.module`](/config/output#outputmodule) 不同，`modern-module` 会完整接管库产物的渲染流程。前者仍沿用常规的 chunk / runtime 渲染流程，再将入口导出转换为 `export` 语句；而 `modern-module` 会直接走一条更适合库产物的专用路径，具体包括：

- 使用独立的 scope hoisting 算法，而不再依赖 [`optimization.concatenateModules`](/config/optimization#optimizationconcatenatemodules)，可以在不引入 Rspack 运行时包装代码的前提下，将多个模块合并到同一个 ESM 产物中。
- 直接将 chunk 渲染为 ESM 文件，并使用真实的 `import` / `export` 语句在 chunk 之间进行链接，即使经过[代码分割](/guide/optimization/code-splitting)，产物依然能被下游工具静态分析。
- 自动对会在多个 chunk 中重复出现的模块进行去重，以减少库被拆分为多个入口时的冗余代码。
- 启用 [`output.library.preserveModules`](/config/output#outputlibrarypreservemodules) 能力：在产物中保留源码目录结构，适合需要“源码一个文件、产物一个文件”发布形态的库。

在配置层面，`modern-module` 也内置了一组默认值，用来简化手动配置：它会自动将 [`output.module`](/config/output#outputmodule) 置为 `true`，将 `output.chunkLoading` 设为 `'import'`，并放宽 [`splitChunks`](/plugins/webpack/split-chunks-plugin) 的默认值，以保证拆分后产物的纯净度。更完整的 ESM 场景和配置说明可参考 [ESM 指南](/guide/features/esm)。

<span id="import-meta-properties"></span>
## 更好的 import.meta 支持 @ahabhgk

在源码侧，Rspack 2.0 继续完善了 `import.meta` 相关能力。对于未知的 `import.meta` 属性，你可以通过 [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) 控制其行为，使它不在编译期被过早静态决定，而是保留到运行时求值。

例如，如果你希望未知的 `import.meta` 属性保留到运行时求值，可以显式配置：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

此外，Rspack 2.0 还补充了对 [`import.meta.main`](/api/runtime-api/module-variables#importmetamain)、[`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename) 和 [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname) 等能力的支持。

<span id="import-defer"></span>
## 更好的延迟导入支持 @ahabhgk

Rspack 在 1.6 首次支持了 `import defer` 语法，到了 2.0，这项能力又向更完整的形态推进了一步。除了最初的 `import defer * as foo from './foo'`，Rspack 还补齐了 `import.defer()` 的支持，使“导入模块但延后执行”的语义能够覆盖更多真实用法。

这让开发者可以更细致地控制模块执行时机和副作用触发时机。

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

<span id="dev-server"></span>
## 更轻量的开发服务器 @chenjiahan

在 2.0 中，`@rspack/dev-server` 不再作为 `@rspack/cli` 的默认依赖，因为并不是所有场景都需要开发服务器。如果你在项目中使用 `rspack dev` 或 `rspack serve`，现在需要手动安装：

```bash
pnpm add -D @rspack/dev-server
```

同时，为了进一步精简 `@rspack/dev-server`，我们引入了 `@rspack/dev-middleware` 来替代 `webpack-dev-middleware`，并用 `connect-next` 替代了 Express v4。

<span id="subpath-alias-import"></span>
## 支持 `#/` 子路径别名导入 @stormslowly

在模块解析方面，Rspack 2.0 支持了 `#/` 形式的子路径别名导入。这样你就可以直接复用 `package.json` 中的 `imports` 字段来组织包内路径映射，而不必额外维护一套独立的别名配置。

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

<span id="mf-share-treeshaking"></span>
## 模块联邦共享依赖 Treeshaking

Rspack 2.0 进一步完善了 [Module Federation](/guide/features/module-federation) 场景下的共享依赖 Treeshaking。它可以对 `shared` 依赖做导出级别的优化，移除实际不会被使用到的部分，从而减少微前端场景中的共享包体积。

在 Rspack 1.x 的模块联邦共享依赖方案里，只要一个依赖被声明到 `shared`，运行时通常就需要准备完整的依赖包。即使应用实际只用到了其中少量导出，也往往会把整份依赖带进来。对于 `antd`、`lodash-es` 这类体积较大、导出较多的库来说，这会明显增加共享包的体积。

Rspack 2.0 将这项能力直接接入了 `ModuleFederationPlugin`。当某个 `shared` 项开启 `treeShaking` 后，Rspack 会为这类共享依赖生成对应的 Tree Shaking 产物，并让运行时优先加载这份更小的结果。如果当前请求无法命中，运行时会回退到完整依赖，以保证行为稳定。

它同时支持 `runtime-infer` 和 `server-calc` 两种模式：

- `runtime-infer` 适合先快速接入和本地验证。运行时会优先复用已经加载过的 Tree Shaking 共享包；如果当前需要的导出无法满足，则自动回退到完整依赖。你也可以通过 `usedExports` 预先声明常用导出，以提升命中率。
- `server-calc` 更适合多应用协作的线上场景。你可以在部署侧统一汇总各个消费者真正会用到的导出，再生成更稳定的共享依赖 Tree Shaking 结果。

对于组件库这类典型场景，这项能力可以把“整包共享”变成“按实际使用结果共享”，减少不必要的代码加载。更完整的配置与工作方式，可以参考 [共享依赖 Tree Shaking 指南](https://module-federation.io/zh/guide/performance/shared-tree-shaking)。

例如：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

<span id="detect-syntax"></span>
### 内置 SWC loader 自动检测语法 @chenjiahan

当你用同一条规则同时处理 `.js`、`.jsx`、`.ts`、`.tsx` 等不同类型文件时，[`builtin:swc-loader` 的 `detectSyntax`](/guide/features/builtin-swc-loader#detectsyntax) 可以根据扩展名自动推断 parser 配置，从而避免重复配置。

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

<span id="resolve-import"></span>
### 支持 CSS parser 的 resolveImport 选项

Rspack 2.0 在 CSS parser 上提供了 [`resolveImport`](/config/module#moduleparsercssautoresolveimport) 选项，`css`、`css/auto`、`css/module` 均可使用。它用于控制构建期是否解析并内联 `@import` at-rule。

当值为 `true`（默认值）时，`@import` 会被视作依赖处理，并用被导入文件的内容替换掉；设置为 `false` 时，`@import` 语句会原样保留在产物中，交由浏览器或下游工具解析。

该选项也可以接收一个函数。Rspack 会对每条 `@import` 规则调用一次，并传入 URL、发起导入的文件路径，以及该 at-rule 上的完整条件（`media`、`supports`、`layer`）。借此你可以按需决定哪些 import 需要内联。例如，可以将本地样式表内联，而把远端 URL 保留为真实的 `@import` 由浏览器加载：

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      'css/auto': {
        resolveImport: ({ url }) => !/^https?:\/\//.test(url),
      },
    },
  },
};
```

<span id="automatic-targets"></span>
### 自动继承 targets 配置 @ahabhgk

[`builtin:swc-loader`](/guide/features/builtin-swc-loader)、[`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader)、[`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin) 和 [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) 的目标环境配置项，现在会默认继承顶层的 [`target`](/config/target) 配置。

这意味着开发者不再需要在 loader、plugin 和顶层目标环境之间手动同步配置，既减少了配置冗余，也能避免“转译目标”和“压缩目标”不一致带来的问题。

<span id="hashed-module-ids-plugin"></span>
### 支持 HashedModuleIdsPlugin

`HashedModuleIdsPlugin` 也在 2.0 中得到支持。如果你希望基于模块的相对路径计算出一个简短、稳定且唯一的 id，可以使用这个插件：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

<span id="enforce-size-threshold"></span>
### 支持 splitChunks.enforceSizeThreshold

代码分割选项 [`splitChunks.enforceSizeThreshold`](/plugins/webpack/split-chunks-plugin) 在 2.0 中得到了支持。它用于设置一个强制拆分的体积阈值：当某个模块组的体积超过该阈值时，[`splitChunks.maxAsyncRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxasyncrequests)、[`splitChunks.maxInitialRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxinitialrequests) 和 `splitChunks.minRemainingSize` 等限制会被忽略，从而确保拆分仍然发生。

如果没有这个配置，请求数限制可能会阻止一个很大的 cache group 被拆分出来，即使拆分本身是更合理的；而加入该阈值后，一旦模块组体积超过阈值，Rspack 就会跳过这些限制并继续完成拆分。

与 webpack 对齐，Rspack 中 `enforceSizeThreshold` 的默认值在生产模式下为 `50000`，其他模式下为 `30000`。同时，它也支持以对象形式针对不同模块类型（如 JavaScript、CSS）分别设置阈值：

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      // 使用数字形式会对所有模块类型生效
      enforceSizeThreshold: 50000,
      // 也可以按模块类型分别配置
      // enforceSizeThreshold: { javascript: 50000, css: 30000 },
    },
  },
};
```

该选项同样可以在 cache group 粒度上配置。你可以保留较严格的全局请求数限制，同时允许某个 cache group（如 `defaultVendors`）在体积超过指定阈值后更积极地完成拆分。

此外，[`minSize`](/plugins/webpack/split-chunks-plugin#splitchunksminsize)、[`maxSize`](/plugins/webpack/split-chunks-plugin#splitchunksmaxsize) 和 [`minChunks`](/plugins/webpack/split-chunks-plugin#splitchunksminchunks) 等与体积和共享度相关的约束仍会继续生效。

<span id="better-tree-shaking"></span>
### 更好的 tree shaking @ahabhgk

2.0 中新增了 `experiments.pureFunctions`，允许通过 `/*#__NO_SIDE_EFFECTS__*/` 注解或 [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) 配置，对跨模块的纯函数调用做更细粒度的分析。

同时，一部分过去没有被正确识别的 CommonJS 导入导出，在 2.0 中也能够被正确解析，从而进一步增强 [tree shaking](/guide/optimization/tree-shaking) 的效果。

- `optimization.removeAvailableModules` 已被移除。
