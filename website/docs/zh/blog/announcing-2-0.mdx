# Rspack 2.0 发布公告

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/rspack-banner-v2.0.png)

我们很高兴地宣布 Rspack 2.0 已正式发布！

## 回顾 1.x

在 2024 年 8 月发布 Rspack 1.0 时，我们设定了一个明确的阶段目标：在保持与 webpack API 和生态兼容的前提下，实现 10 倍的构建性能提升。

回顾 1.x 阶段，这一目标已经基本达成。Rspack 不仅实现了 webpack 的核心能力和插件 API，也在开发体验、产物优化以及现代语言特性支持等方面不断演进，陆续引入并完善了增量构建、按需编译、持久化缓存、常量内联、虚拟模块、Barrel 文件优化等新特性。

同时，Rspack 也在被更多用户采用，与 1.0 发布时相比，**Rspack 的周下载量已由 10 万增长至 500 万**：

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

我们还围绕 Rspack 打造了 Rstack，一套统一的 JavaScript 工具链，包括 [Rsbuild](https://github.com/web-infra-dev/rsbuild)、[Rslib](https://github.com/web-infra-dev/rslib)、[Rstest](https://github.com/web-infra-dev/rstest)、[Rspress](https://github.com/web-infra-dev/rspress)、[Rsdoctor](https://github.com/web-infra-dev/rsdoctor) 和 [Rslint](https://github.com/web-infra-dev/rslint)。这些工具面向不同的场景，但都共享了同一个使命：让 Web 开发变得更加简洁、一致和高效。

Rspack 也在逐步融入更广泛的 JavaScript 生态，社区中许多优秀的工具、框架和平台已经提供对 Rspack 的支持，包括但不限于：[Angular Rspack](https://nx.dev/docs/technologies/angular/angular-rspack)、[Addfox](https://github.com/addfox/addfox)、[Docusaurus](https://github.com/facebook/docusaurus)、[Extension.js](https://github.com/extension-js/extension.js)、[ice.js](https://github.com/alibaba/ice)、[Lynx](https://github.com/lynx-family/lynx)、[Meteor](https://github.com/meteor/meteor)、[Modern.js](https://github.com/web-infra-dev/modern.js)、[Next.js](https://github.com/vercel/next.js)、[Nuxt](https://github.com/nuxt/nuxt)、[Nx](https://github.com/nrwl/nx)、[Kmi](https://github.com/kmijs/kmi)、[Re.Pack](https://github.com/callstack/repack)、[Remotion](https://github.com/remotion-dev/remotion)、[Shakapacker](https://github.com/shakacode/shakapacker)、[Storybook](https://github.com/storybookjs/storybook)、[TanStack Router](https://tanstack.com/router)、[Unplugin](https://github.com/unjs/unplugin)、[Vize](https://github.com/ubugeeei/vize)、[xmcp](https://github.com/basementstudio/xmcp)、[Zephyr](https://zephyr-cloud.io/)。在此，我们衷心感谢所有支持 Rspack 的社区项目和团队。

## 迈向 2.0

JavaScript 生态在持续蓬勃发展，coding agent 正在改变软件的研发范式。这促使我们重新思考，一个面向未来的打包工具应当如何演进。

**Rspack 的目标不只是成为一个"更快的 webpack"**。在 1.x 阶段，我们有意让 Rspack 的 API 和默认值与 webpack 5 保持一致，这一策略是为了帮助现有项目低成本迁移到 Rspack。但随着 JavaScript 模块规范和生态的发展，一些历史设计已不再适合作为当下的默认选择。

从 2.0 开始，在保持对 webpack 生态兼容的前提下，Rspack 将逐步引入更符合现代 JavaScript 开发的默认行为、API 设计和构建产物。这将是一个渐进的过程，我们会尽量避免在单个主版本中引入过多不兼容变更。同时也会提供迁移指南与 Agent Skills，将迁移成本控制在可接受范围内。

## 性能提升

### 构建提速 \{#build-performance}

构建性能始终是 Rspack 的核心指标之一。与 Rspack 1.7 相比，Rspack 2.0 的整体性能提升约 10%，相较 1.0 最多可提升 100%。

| 版本       | 生产构建（无缓存） | 生产构建（有缓存） | 热更新     |
| ---------- | ------------------ | ------------------ | ---------- |
| Rspack 1.0 | **5.6 s**          | **5.6 s**          | **128 ms** |
| Rspack 1.7 | **3.6 s**          | **2.2 s**          | **134 ms** |
| Rspack 2.0 | **3.1 s**          | **1.4 s**          | **118 ms** |

> 数据来源：[rspack-react-10k-benchmark](https://github.com/LingyuCoder/rspack-react-10k-benchmark/actions/runs/24710780593)

这些改进主要来自对核心架构的持续优化。我们针对关键性能路径重构了部分算法与数据结构，升级了部分过时依赖，并清理了不再使用的代码路径。

在开启 [持久化缓存](/config/cache#persistent-cache) 的场景下，构建性能和内存占用还可进一步改善：

- **热启动性能提升 50%+**：现在 Rspack 支持对 [SWC 压缩插件](/plugins/rspack/swc-js-minimizer-rspack-plugin) 的结果进行缓存复用
- **内存占用降低 20%+**：这得益于底层存储实现的优化，启用缓存时的内存开销将显著降低

### 精简默认依赖 \{#fewer-deps}

Rspack 2.0 减少了默认安装的 npm 依赖数量：

- `@rspack/dev-server`：**依赖数量从 192 降至 1**。
- `@rspack/core`：**依赖数量从 8 降至 1**。
- `@rspack/cli`：**现在为零依赖**。

在 Rspack v1 中，`@rspack/dev-server` 通过 `webpack-dev-server` 间接引入了较多依赖，这增加了安装体积，也提高了依赖管理的复杂度。为此，我们对它进行了重构，梳理并精简了功能和依赖，**使安装体积减少超过 90%**（从 15 MB 降至 1.4 MB）。

同时，`@rspack/cli` 也不再默认依赖 `@rspack/dev-server`，这意味着如果只使用构建能力，将不再需要引入开发服务器相关的依赖。

此外，我们还采用了以下方式来减少依赖：

- 将非核心依赖调整为可选依赖。例如 [@module-federation/runtime-tools](https://www.npmjs.com/package/@module-federation/runtime-tools) 现在仅在使用 [模块联邦插件](/plugins/webpack/module-federation-plugin) 时才需要手动安装。
- 使用更轻量的替代实现，例如以 [connect-next](https://github.com/rstackjs/connect-next) 替代 Express。对于开发服务器而言，Connect 的中间件模型已经能够覆盖大多数场景，这也与 Rsbuild 的开发服务器实现保持一致。
- 将部分依赖 [打包进发布产物](https://e18e.dev/blog/bundling-dependencies)，由 Rspack 统一控制这些依赖及其子依赖的版本，避免依赖自动升级带来的潜在供应链风险。
- 优先使用 Node.js 20+ 内置的原生 API，例如用 [styleText](https://nodejs.org/api/util.html#utilstyletextformat-text-options) 替代 `picocolors`。

## 产物优化

### 静态分析增强 \{#better-tree-shaking}

Rspack 2.0 在静态分析能力上进行了增强，使更多复杂代码模式可以被 [tree shaking](/guide/optimization/tree-shaking) 优化。一些过去难以分析的场景，现在也能够参与导出级别的裁剪。

- **CommonJS `require` 解构赋值**：Rspack 现在可以识别解构中实际使用到的导出成员，仅保留必要代码

```js
const { bar } = require('./foo');
```

- **CommonJS `require` 成员访问与调用**：Rspack 现在能继续向下分析对象的成员访问和调用，判断哪些导出被使用

```js
const foo = require('./foo');

foo.bar;
foo.baz();
```

- **动态 `import()` 结果的内联成员访问与调用**：对于这种直接在表达式中访问模块成员的写法，Rspack 同样可以识别使用到的导出，并进行裁剪

```js
(await import('./foo')).bar;
```

### 编译器注解支持 \{#pure-functions}

Rspack 2.0 现在支持 [编译器注解](https://github.com/javascript-compiler-hints/compiler-notations-spec)，允许你使用 `#__NO_SIDE_EFFECTS__` 将一个函数标记为无副作用，如果该函数被调用后的返回值没有被使用，tree shaking 会自动移除未使用的代码。

例如，在下面的代码中，`join` 被标记为一个无副作用函数。当它的返回值没有被使用时，这次调用会被安全移除。

```js title="utils.js"
/*#__NO_SIDE_EFFECTS__*/
export function join(a, b) {
  return `${a}-${b}`;
}
```

```js title="index.js"
import { join } from './utils';

join('btn', 'primary');
```

> 该功能仍处于实验阶段，需要通过 `experiments.pureFunctions` 启用，查看 [指南](/guide/optimization/tree-shaking#no_side_effects-注解) 了解更多。

对于无法直接修改源码的第三方模块，Rspack 还允许你通过 [pureFunctions](/config/module#moduleparserjavascriptpurefunctions) 选项手动声明纯函数列表，以达到同样的效果。

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      javascript: {
        pureFunctions: ['myFunctionName'],
      },
    },
  },
};
```

### 模块联邦 tree shaking \{#mf-share-treeshaking}

Rspack 现在支持对 [模块联邦](/guide/features/module-federation) 的共享依赖进行 tree shaking。它可以在导出级别裁剪共享依赖，移除未使用的部分，从而减小共享包的体积。

在 Rspack 1.x 中，只要依赖被声明为 `shared`，运行时通常需要加载完整包。即使只用到少量导出，也会引入全部内容。在共享体积较大的库时，这会带来显著的开销。

而 Rspack 2.0 支持在 `shared` 选项中开启 `treeShaking`。此时，`ModuleFederationPlugin` 会为共享依赖生成裁剪后的产物，运行时优先加载该结果；若无法命中，再回退到完整依赖，以保证行为一致。

例如：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

> 查看 [共享依赖 tree shaking 指南](https://module-federation.io/zh/guide/performance/shared-tree-shaking) 了解更多。

## 改进 ESM 支持

### 纯 ESM 包 \{#pure-esm}

Rspack 的核心包现在以 pure ESM 包的形式发布，并移除了自身的 CommonJS 构建产物，这使模块加载方式更加统一，也更符合当前 Node.js 的主流实践。

本次变更涉及以下 npm 包：

- [@rspack/core](https://www.npmjs.com/package/@rspack/core)
- [@rspack/cli](https://www.npmjs.com/package/@rspack/cli)
- [@rspack/dev-server](https://www.npmjs.com/package/@rspack/dev-server)
- [@rspack/plugin-react-refresh](https://www.npmjs.com/package/@rspack/plugin-react-refresh)

在 Node.js 20 及以上版本中，运行时已原生支持通过 [require(esm)](https://nodejs.org/api/modules.html#loading-ecmascript-modules-using-require) 加载 ESM 模块。因此，对大多数仍通过 JavaScript API 使用 Rspack 的项目来说，这一变更通常不会带来实际影响，也无需额外修改现有代码。

### import.meta 支持 \{#import-meta}

Rspack 2.0 改进了对 [import.meta](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/import.meta) 的支持。

在 Rspack 1.x 中，为了兼容非 ESM 产物，Rspack 会在编译阶段解析 `import.meta` 并替换为对应的值。对于无法识别的 `import.meta` 属性，通常会直接替换为 `undefined`。

从 Rspack 2.0 开始，在生成 ESM 产物时，Rspack 默认会保留无法识别的 `import.meta` 属性，而不是在编译阶段将其替换，这允许你使用一些自定义 `import.meta` 属性，也使其行为更符合 ESM 规范。

这个行为也可以通过 [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) 进行控制，例如：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

此外，Rspack 2.0 还补充了对 [`import.meta.main`](/api/runtime-api/module-variables#importmetamain)、[`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename) 和 [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname) 的支持。

### import defer 支持 \{#import-defer}

[import defer](https://github.com/tc39/proposal-defer-import-eval) 是 JavaScript 中一项用于延后模块求值的能力，也已经在 [TypeScript 5.9](https://devblogs.microsoft.com/typescript/announcing-typescript-5-9-beta/#support-for-import-defer) 中得到支持。它允许你在导入模块时先完成加载，而不立即执行模块及其依赖，从而更好地控制代码加载和副作用发生的时机。

Rspack 从 1.6 开始支持 `import defer * as foo from './foo'` 语法。在 2.0 中，我们进一步补齐了对 `import.defer()` 的支持，使这项能力能够覆盖更多实际场景。

- 启用实验性开关：

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

- 使用 `import.defer()` 来动态导入模块：

```js title="app.js"
const file = Math.random() > 0.5 ? 'a.js' : 'b.js';

import.defer('./dir' + file);
```

### 改进 ESM 库构建 \{#modern-module}

在构建一个 JavaScript 库时，ESM 输出的质量会直接影响下游工具的分析和优化效果。更纯净的 ESM 产物，通常也更有利于静态分析、代码分割和 tree shaking。

在 Rspack 2.0 中，你可以将 [`output.library.type`](/config/output#outputlibrarytype) 配置为 `'modern-module'`，以生成更适合库发布的 ESM 产物。

```js title="rspack.config.mjs"
export default {
  output: {
    library: {
      type: 'modern-module',
    },
  },
};
```

相比面向普通 ESM 输出的 [`output.module`](/config/output#outputmodule) 选项，`modern-module` 针对库构建场景做了专门优化。它基于 Rspack 1.x 中引入的实验性 [`rspack.experiments.EsmLibraryPlugin`](https://v1.rspack.rs/plugins/rspack/esm-library-plugin) 演进而来，生成的产物更便于下游工具继续分析和处理。在代码分割场景下，它也能保持更清晰的 ESM 结构，并减少多入口构建中的重复代码。

此外，`modern-module` 还支持保留源码目录结构，详见 [ESM 指南](/guide/features/esm)。

## 新特性

### React Server Components 支持 \{#rsc-support}

React Server Components（简称 RSC）是 React 提供的一种组件模型：它允许一部分组件直接在服务端执行和渲染，再把结果传递给客户端；与此同时，需要交互的部分仍然可以作为 Client Components 在浏览器中运行。

这种模型可以让数据获取更贴近服务端，减少发送到客户端的 JavaScript 体积，并更自然地把“服务端渲染内容”和“客户端交互能力”组合在同一个应用里。因此，RSC 也正在成为 React 全栈框架中越来越重要的一类基础能力。

Rspack 2.0 内置了对 React Server Components 通用的底层构建能力的支持，包括：

1. 编译期检查：在编译阶段拦截违反 RSC 规范的 React API 调用，提前规避运行时错误。
2. CSS 支持：在编译阶段收集服务端与客户端组件样式，并在渲染时注入。
3. HMR 支持：同时支持服务端和客户端组件的热更新。

你可以通过如下两种方式来使用它：

- **使用 Rsbuild**：Rsbuild 提供对 React Server Components 的开箱即用支持，可以快速创建一个 RSC 项目。详见 [rsbuild-plugin-rsc](https://github.com/rstackjs/rsbuild-plugin-rsc)。
- **手动配置 Rspack**：你也可以参考 [React Server Components 文档](https://v2.rspack.rs/guide/tech/rsc)，手动添加 React Server Components 相关配置。

[rspack-rsc-examples](https://github.com/rstackjs/rstack-examples/tree/main/rspack/rspack-rsc) 仓库中提供了使用 Rspack 构建 RSC 应用的完整示例。

生态落地与合作：

- Modern.js 框架目前已采用 Rspack 的 RSC 实现（[详见 Modern.js RSC 文档](https://modernjs.dev/guides/basic-features/render/rsc.html)）。
- 支持 React Router Data Mode（[React Router 示例](https://github.com/rstackjs/rsbuild-plugin-rsc/tree/main/examples/react-router)）。
- 我们正与 [TanStack](https://tanstack.com/) 展开深度合作，全面支持其利用 Rspack 作为底层引擎来编译和构建 RSC 应用。

### `#/` 子路径别名导入 \{#subpath-alias-import}

在模块解析方面，Rspack 2.0 支持了 `#/` 形式的子路径别名导入。这样你就可以直接复用 `package.json` 中的 `imports` 字段来组织包内路径映射，而不必额外维护一套独立的别名配置。

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

### 简化目标环境配置 \{#automatic-targets}

在 Rspack 1.x 中，顶层的 [target](/config/target) 选项不会影响 loader 或压缩工具的转换目标，你通常还需要在 loaders 或压缩插件的选项中分别配置目标环境选项。这带来了额外的配置成本，同一份目标环境信息需要在多个位置重复声明。

Rspack 2.0 对此进行了改进，内置的 loaders 和压缩插件会默认解析顶层的 `target` 配置，并据此推断各自需要的目标环境设置。多数情况下，你只需要在顶层声明一次目标环境，就能让 JavaScript 代码转换、CSS 转换和代码压缩保持目标一致。

```js title="rspack.config.mjs"
export default {
  target: 'browserslist: Chrome >= 100',
  module: {
    rules: [
      {
        test: /\.js$/,
        use: [
          {
            loader: 'builtin:swc-loader',
            options: {
              env: { targets: 'Chrome >= 100' }, // [!code --]
            },
          },
        ],
      },
    ],
  },
  optimization: {
    minimizer: [
      new rspack.SwcJsMinimizerRspackPlugin(),
      new rspack.LightningCssMinimizerRspackPlugin({
        minimizerOptions: { targets: 'Chrome >= 100' }, // [!code --]
      }),
    ],
  },
};
```

> 查看 [target - 目标环境继承](/config/target#目标环境继承) 了解更多。

### 简化 swc-loader 配置 \{#detect-syntax}

在以往的配置中，如果希望 [builtin:swc-loader](/guide/features/builtin-swc-loader) 同时处理 `.js`、`.jsx`、`.ts`、`.tsx` 等文件，需要根据文件类型显式指定 `jsc.parser` 中 的 `syntax`、`jsx`、`tsx`等选项，以确保 SWC 能以正确的语义解析代码，例如将 `<` 解析为 JSX 语法还是 TypeScript 泛型。

这带来了一定的配置复杂度。为了让 SWC 正确解析不同后缀的文件，通常需要定义多条 `rules`，导致配置变得冗长，不利于维护。

为了解决这一问题，Rspack 2.0 在内置的 `swc-loader` 中引入了 [detectSyntax](/guide/features/builtin-swc-loader#detectsyntax) 选项。启用 `detectSyntax: 'auto'` 后，loader 会根据文件扩展名自动推断 `syntax`、`jsx`、`tsx` 等 `parser` 选项。通过一条规则即可覆盖不同的文件类型，使配置更清晰。

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

### 支持控制 CSS 导入 \{#resolve-import}

Rspack 2.0 的 CSS 解析器提供了 [`resolveImport`](/config/module#moduleparsercssautoresolveimport) 选项，它用于控制构建时是否解析并内联 `@import`。

默认情况下，Rspack 会解析 `@import`，并将导入的内容直接合并到当前文件中。你也可以将 `resolveImport` 设为 `false`，保留原始的 `@import`，交由浏览器或下游工具处理。

`resolveImport` 还支持传入函数，用来按需决定每一条 `@import` 是否需要内联。例如，你可以只内联本地 CSS，而将远程地址保留为 `@import`：

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      'css/auto': {
        resolveImport: ({ url }) => !/^https?:\/\//.test(url),
      },
    },
  },
};
```

### 使用哈希作为模块 ID \{#hashed-module-ids-plugin}

Rspack 2.0 为 [optimization.moduleIds](/config/optimization#optimizationmoduleids) 新增了 `hashed` 选项。启用后，Rspack 会基于模块路径生成较短且稳定的哈希值，并将其用作模块 ID。

这适用于希望在保持模块 ID 稳定性的同时，进一步缩短模块 ID 长度的场景。对于模块数量较多的大型应用，这在某些情况下也有助于减少产物中的 ID 体积。

```js title="rspack.config.mjs"
export default {
  optimization: {
    moduleIds: 'hashed',
  },
};
```

### 代码分割改进 \{#enforce-size-threshold}

Rspack 2.0 现在支持 `splitChunks.enforceSizeThreshold` 选项，用于为代码分割设置一个强制生效的体积阈值。

默认情况下，`splitChunks` 中的请求数限制选项，例如 [maxAsyncRequests](/plugins/webpack/split-chunks-plugin#splitchunksmaxasyncrequests) 和 [maxInitialRequests](/plugins/webpack/split-chunks-plugin#splitchunksmaxinitialrequests)，可能会阻止较大的 chunk 继续拆分。在启用 `enforceSizeThreshold` 后，只要模块组的体积超过设定阈值，Rspack 就会忽略这些限制并进行强制拆分。

在生产模式下，Rspack 默认为所有模块类型设置 `50000`（500 kB）的 `enforceSizeThreshold`，在其他模式下默认值为 `30000`。你也可以按自定义该选项：

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      // 对所有模块类型生效
      enforceSizeThreshold: 50000,
      // 也可以按模块类型分别配置
      // enforceSizeThreshold: { javascript: 50000, css: 30000 },
    },
  },
};
```

这个选项也支持在 cache group 中单独配置，这允许你为特定的 cache group 指定单独的值。

## 后续方向

Rspack 2.0 不是终点，而是一条新路线的开始。

在 2.0 中，我们已经开始为 Rspack 引入一些更面向未来的设计。未来，我们会沿着这些方向继续推进：进一步提升产物优化能力与构建链路效率，完善面向 AI 场景的问题分析和迁移支持，并持续吸收新的 Web 平台能力，更好地支撑上层工程体系。

### 更优的产物优化

Rspack 和 webpack 一样，在很多默认优化上都会优先保证产物行为的稳定性，尤其是在 side effects 判断等环节通常会更保守。这种策略适合大多数项目，但我们也注意到，对于追求极致产物体积的场景来说，仍然有进一步挖掘的空间。

Rspack 2.0 已经支持基于 pure annotation 的更细粒度优化。后续会继续提供更高级的 Tree Shaking 能力和更可控的优化配置，让业务可以按场景选择产物优化级别，在安全默认值与更激进优化之间取得平衡。

我们也会结合静态类型信息增强优化分析，借鉴 Prepack、Closure Compiler 和 React Compiler 的思路，把更多来自业务代码的可优化信息引入 bundler。

代码拆分也会走向更细粒度。今天的 code splitting 最小粒度通常仍然是模块，模块内部的局部复用仍可能带来额外加载。我们会参考 esbuild、Turbopack 等工具在 module splitting 方向上的经验，按照实际复用关系把公共模块拆成更小的子模块，在保证缓存稳定性、调试体验和运行时行为可控的前提下减少不必要加载；也会结合 `import defer` 等模块能力探索新的加载策略。

### 更优的构建性能

Rspack 会继续优化自身的算法、数据结构以及增量构建路径，但我们也越来越明确地看到：在很多真实项目里，瓶颈并不总是出在 bundler 主流程本身，而更常见地出现在 Tailwind、lint、typecheck 等社区插件和配套工具上。

因此，后续的性能工作不会只关注 Rspack 内核。我们也会继续推动 Rstack 生态中的配套工具演进，例如在 Rslint 里探索基于 tsgo 的 lint 和 typecheck 方案，作为当前常见的 `eslint-webpack-plugin` 和 `fork-ts-checker-webpack-plugin` 组合的替代选择，让整个构建链路都能获得更稳定的性能收益。

### 更好的 AI 集成

越来越多的项目已经开始使用 AI 工具处理 Rspack 相关问题，包括构建性能分析、产物优化和自动化迁移。我们已经通过 [Rstack Skills](https://github.com/rstackjs/agent-skills) 为这类工作提供了一定支持。同时，从实际使用中我们也越来越清楚：AI 是否真正能够帮助解决构建问题，很大程度上取决于三个前提，即是否能够获得足够准确的上下文、是否能够稳定复现问题，以及是否能够以足够低的成本快速验证假设。

接下来，这部分工作会主要围绕四个方向展开：

- 更好的 Debugger 支持，让 AI Agent 能够更容易基于 Rspack 的调试版本排查问题，并把编译期与运行时的信息关联起来。
- 更丰富且更结构化的诊断信息，在编译阶段暴露更多可机器消费的过程数据，例如 Tree Shaking、scope hoisting 等优化阶段的 bailout reason，以及更清晰的模块、chunk 和插件链路信息，帮助 AI 自动分析并迭代优化。
- 更稳定的上下文采集与问题复现能力，例如更清晰地暴露配置解析结果、环境信息和关键编译路径，让 AI 生成的建议更容易被复现和验证。
- 更快的反馈循环，为 Agent 场景探索更高效的编译模式，缩短“提出假设、执行验证、继续迭代”的闭环时间。

### 更现代化的产物格式

在 1.x 和 2.0 阶段，Rspack 一直兼容 webpack 的产物格式。这对迁移正确性和生态兼容非常重要，但也意味着我们需要继承一部分历史包袱。为了同时支持 CJS、UMD、ESM 等不同输出格式，以及 HMR、Module Federation 等运行时能力，传统 bundler 产物通常会在可读性和二次静态分析能力上做出妥协。

未来我们会在保留关键运行时能力的前提下，继续探索更现代的产物形态，例如更纯净的 ESM 输出，以及基于 import maps、`import defer`、module fragments 等新标准的组织方式，让产物既更适合运行，也更适合被下游工具继续分析和优化。

### 持续跟进社区规范与平台能力

JavaScript 社区仍在持续演进，新的语法、模块能力和运行时接口也在不断成熟。Rspack 会继续第一时间跟进这些变化，把对新规范的支持尽可能早地带给上层工具和框架。2.0 中已经可以看到这一方向，例如对 pure ESM、`import.meta` 和 `import defer` 的持续完善；后续我们会继续沿着这条路线推进。

### 更丰富的能力支持

今天的 bundler 已经不只是把模块打成 bundle 的工具。对于框架和上层工具来说，它还需要承担多环境编译、服务端与客户端边界分析、样式与资源收集、运行时协调等更底层的职责。

Rspack 2.0 已经在 React Server Components 上迈出了一步。接下来，我们会继续把更多能力沉淀为稳定的框架原语，而不是把复杂度全部留给框架层自行实现。这样，上层框架和工具在接入 Rspack 时，可以用更少的适配代码获得更完整的默认能力。

## 常见问题 \{#faq}

### 2.0 以后，Rspack 是否不再兼容 webpack？

不是。Rspack 仍会尽量保持与 webpack 的兼容性，持续帮助现有项目平滑迁移和使用。

变化在于，从 2.0 开始，Rspack 会在继续重视兼容性的同时，更积极地推进构建性能、产物体积、社区规范和 Agent 支持等方向的演进。当更合适的方案需要对部分历史行为做出调整时，我们会在充分评估影响后稳步推进。

即便如此，我们仍会尽量提供清晰的过渡方式、升级说明和迁移方案，帮助用户理解这些变化，并以可控的成本完成迁移。

### 2.0 发布后，1.x 版本是否会完全停止维护？

不会。2.0 发布后，Rspack 的主要演进方向会逐步转向 2.x，但这并不意味着 1.x 会立刻完全停止维护。对于仍在使用 1.x 的项目，我们会根据实际情况，在一段较长的时间内继续提供必要的维护，例如关键问题修复或迁移期支持。

但从长期来看，新增能力和主要优化都会优先在 2.x 上推进，因此如果你的项目还停留在 1.x，我们建议尽早规划升级。
