# Rspack 2.0 发布公告

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

距离 Rspack 1.0 正式发布已经过去了 1 年 8 个月。在这期间我们保持着每周一个小版本，每两个月一个中版本的节奏，持续补齐 webpack 的功能和生态兼容性，让更多项目能够以更低成本迁移到 Rspack。我们逐步完善了模块能力、开发服务器、缓存体系、ESM 支持以及更多工程化能力，使 Rspack 成为一个兼容 webpack 生态的同时具备更高构建性能的打包方案。

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

![Rspack 2.0 repo stats](https://assets.rspack.rs/rspack/assets/rspack-v2-0-repo-stats.png)

## 尝试 2.0

### 新用户安装

如果你是新用户，推荐直接创建一个基于 2.0 的新项目开始体验。需要注意的是，Rspack 2.0 要求 **Node.js 20.19+** 或 **22.12+**。

如果你希望获得一套开箱即用、默认由 Rspack 驱动的构建体验，我们推荐直接使用 Rsbuild 创建项目：

```bash
npm create rsbuild@latest
```

如果你更希望直接使用 Rspack CLI，也可以执行：

```bash
npm create rspack@latest
```

然后按照提示完成初始化即可。更多信息也可以参考 [快速上手](/guide/start/quick-start) 文档。

### 老用户升级

Rspack 2.0 作为一个大版本，在带来了不少新特性的同时，也移除了诸多历史上废弃的功能，因此建议参考的 [迁移指南](/guide/migration/rspack_1.x)，其中已经整理了从 v1 到 v2 的主要不兼容变更和对应迁移方式。

如果你在使用支持 Skills 的 Coding Agent，也推荐直接用 AI 辅助迁移。按照迁移文档里的 Agent Skills 说明，安装 rspack-v2-upgrade 技能后，就可以让 Agent 结合你的项目配置和官方迁移规则完成升级流程。对于配置项较多、历史包袱较重的项目，这会比手工逐项排查更高效。

```
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## 后续方向

Rspack 2.0 不仅是一个里程碑，也是一个新的起点。

长期来看， Rspack 团队将把工作重心从 “兼容 Webpack” 转向在**构建性能**、**产物体积**、**社区规范**、**Agent 协同**等方向，配合 Rstack 内一整套开发工具，为开发者提供 “AI 友好的一体化 JavaScript 工具链”。

短期来看，我们将着眼于如下工作：

- 持续改进 tree shaking 和 side effects 分析能力，并尝试探索 module splitting 方案，以显著降低产物体积
- 持续优化构建链路上各个环节的算法和数据结构，并将它们接入到持久化缓存当中，以提升各种场景下的构建速度
- 持续增强 ESM 产物的质量和稳定性，并与社区其他技术团队共同探究 bundler 对于 ESM 规范更好的支持方式
- 持续建设 Rstack Skills，为开发者提供更多 AI 友好的工具来提升开发效率，降低问题排查、产物分析的成本

## 常见问题

### 2.0 以后，Rspack 是否不再兼容 webpack？

不是。Rspack 后续仍会保持有限的 webpack 兼容性，并尽可能让现有项目能够继续平滑迁移和使用。只是从 2.0 开始，Rspack 不会再把“兼容”作为唯一目标；当我们在构建性能、产物体积、社区规范、Agent 支持等方向上找到了更适合的方案，并且这些方案必须打破一部分历史兼容时，我们会选择做出相应调整。

但即使如此，我们仍会尽量提供详细的过渡方式、升级说明和迁移方案，帮助用户理解这些变化，并以可控的成本完成迁移。

### 2.0 发布后，1.x 版本是否会不再维护？

不会。2.0 发布后，Rspack 的主要演进方向会逐步转向 2.x，但这并不意味着 1.x 会在发布后立刻完全停止维护。对于仍在使用 1.x 的项目，我们会根据实际情况在一个较长的时间段内提供必要维护，例如关键问题修复或迁移期支持。

但从长期来看，新增能力和主要优化都会优先在 2.x 上推进，因此如果你的项目还停留在 1.x，我们仍然建议尽早规划升级。

<span id="rsc-support"></span>
## 内置 React server component @SyMind

Rspack 2.0 提供了内置的 [React Server Components](/guide/tech/rsc) 支持。它不再完全依赖上层框架做额外拼装，而是在核心中直接提供了面向 RSC 的插件与编译支持：通过 [`rsc.createPlugins()`](/guide/tech/rsc#配置插件) 生成的 `ServerPlugin` 与 `ClientPlugin`，Rspack 可以协调 Server Compiler 和 Client Compiler 两套构建流程，自动收集 `"use client"`、`"use server"`、`"use server-entry"` 等指令信息，生成 RSC 运行时所需的 manifest，并在服务端组件变更时通过 [`onServerComponentChanges`](/guide/tech/rsc#开发服务器) 暴露更新信号。

与此同时，`builtin:swc-loader` 也被增强为可以识别并转换这些 RSC 指令，使 React 19 时代的 RSC、Server Actions、客户端引用和服务端引用能够在同一套构建体系里协同工作。

例如：

```js title="rspack.config.mjs"
import { experiments } from '@rspack/core';

const { createPlugins } = experiments.rsc;
const { ServerPlugin, ClientPlugin } = createPlugins();

export default [
  {
    target: 'web',
    plugins: [new ClientPlugin()],
  },
  {
    target: 'node',
    plugins: [new ServerPlugin()],
  },
];
```

## 更快的构建速度

### 内部实现优化

![Rspack 2.0 performance comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-perf-comparison.png)

Rspack 团队一直非常重视性能，在 1.x 版本中我们实现了持久化缓存、增量构建、原生文件系统监听器、lazy compilation、barrel 文件优化等诸多能力，使得各种场景的构建性能相对于 1.0 版本都有大幅提升。而在 2.0 版本中，我们又进一步优化了内部数据结构和算法，在此前的基础上又带来约 10% 的性能收益。

<span id="portable-cache"></span>
### 可移植缓存 @jerrykingxyz

在缓存场景下，我们还带来了 Portable Cache，让缓存可以跨不同机器、不同路径甚至不同环境复用，从而更适合团队协作、远程缓存以及后续可能的分布式构建场景。

你可以在 Rspack 2.0 中通过 [`cache.portable`](/config/cache#cacheportable) 开启它：

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

除了可移植缓存之外，Rspack 2.0 也提供了 [`cache.readonly`](/config/cache#cachereadonly) 选项。开启后，缓存只会从磁盘读取而不会写回，这对于 CI 场景尤其有用，可以直接复用预热缓存而不修改它。

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

## 面向 ESM 规范

<span id="pure-esm"></span>
### 纯 ESM 包 @chenjiahan

Rspack 2.0 也让自身产物更加符合 ESM 规范。`@rspack/core`、`@rspack/cli`、`@rspack/dev-server` 和 `@rspack/plugin-react-refresh` 等包被正式切换为 [Pure ESM](/guide/migration/rspack_1.x#pure-esm-包) 发布，并移除了 CommonJS 构建产物。这使 Rspack 的包形态与现代 Node.js / ESM 生态保持一致，简化了模块加载和工具链集成方式。

因此，你需要将 Node.js 升级到原生支持 `require(esm)` 的 20.19+ 或 22.12+ 版本。

<span id="better-esm"></span>
### 更好的 ESM 产物 @jserfeng

针对库打包场景，ESM 产物质量尤为重要，纯净的产物更易于静态分析，从而在库被引用时能更易被代码切分和摇树优化。Rspack 2.0 在之前推出的实验性的 [`EsmLibraryPlugin`](/guide/migration/rspack_1.x#移除-esmlibraryplugin) 的基础上进行了一系列的优化，将 ESM 产物质量进一步提升，同时将配置简化为更自然的 [`modern-module`](/config/output#outputlibrarytype) 和 [`output.module`](/config/output#outputmodule) 上。

你可以在库构建中这样配置：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
    library: {
      type: 'modern-module',
    },
  },
};
```

<span id="import-meta-properties"></span>
### Import meta 属性 @ahabhgk 目前绝大多数项目源码是基于 ESM 规范开发，在 Rspack
2.0 中调整了对未知 `import.meta` 属性的处理方式，可以通过
[`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta)
配置使得 `import.meta.*` 可以不在编译期被过早静态决定，而是保留到运行时求值。

例如，如果你希望未知的 `import.meta` 属性保留到运行时求值，可以显式配置：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

此外，Rspack 2.0 还补充了对 [`import.meta.main`](/api/runtime-api/module-variables#importmetamain)、[`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename) 和 [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname) 等的支持。

<span id="import-defer"></span>
### 延迟 import @ahabhgk

Rspack 在 1.6 首次支持了 `import defer` 语法，而在 2.0 中，这项能力继续向更完整的形态推进。除了最初的 `import defer * as foo from './foo'`，Rspack 后续还补齐了 `import.defer()` 的支持，使得“导入模块但延后执行”的语义可以覆盖更多真实用法。这让开发者可以更细致地控制模块执行时机和副作用触发时机。

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

<span id="dev-server"></span>
## 更轻量的开发服务器 @chenjiahan

[`@rspack/dev-server`](/guide/migration/rspack_1.x#开发服务器) 在 2.0 中将不再作为 `@rspack/cli` 的默认依赖，而是变成可选安装项；同时，它也在内部中间件栈上引入了 `@rspack/dev-middleware`，并用 `connect-next` 取代了过去更重的 Express v4 方案。

调整后的结果是：开发服务器更轻量，也更适合作为独立能力被上层框架按需集成。关于常规使用方式，可以参考 [Rspack Dev Server 指南](/guide/features/dev-server)。

如果你在项目中使用 `rspack dev` 或 `rspack serve`，现在需要显式安装 `@rspack/dev-server`：

```bash
pnpm add -D @rspack/dev-server
```

<span id="module-resolution"></span>
## 更好的模块解析 @stormslowly

<span id="subpath-alias-import"></span>
### `#/` 子路径别名导入

在模块解析方面，Rspack 2.0 还支持了 `#/` 形式的子路径别名导入。这样一来，你可以直接复用 `package.json` 里的 `imports` 字段来组织包内路径映射，而不必额外维护一套独立的别名配置。

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

<span id="pnp-manifest"></span>
### resolve.pnpManifest

在 Yarn PnP 相关场景下，Rspack 2.0 的 resolver 也补充了 `resolve.pnpManifest` 选项支持，用于显式指定 PnP 清单文件路径。这样在更复杂的包管理和 monorepo 环境下，你可以更细致地控制 PnP 元信息的读取与解析行为，而不必完全依赖自动探测。

```js title="rspack.config.mjs"
import path from 'node:path';

export default {
  resolve: {
    pnp: true,
    pnpManifest: path.resolve(import.meta.dirname, '.pnp.cjs'),
  },
};
```

## 更好的模块联邦支持

<span id="mf-runtime"></span>
### 模块联邦运行时改进 @chenjiahan

`@module-federation/runtime-tools` 也在 2.0 中从直接依赖改成了可选 peer dependency，并扩展了兼容版本范围以同时接受较新的运行时版本。

这些调整让依赖边界更清晰，也让上层框架和工具在集成时拥有更高的灵活度。

例如，如果你使用 `ModuleFederationPlugin`，现在需要显式安装 `@module-federation/runtime-tools`：

```bash
pnpm add @module-federation/runtime-tools
```

<span id="mf-share-treeshaking"></span>
### 模块联邦共享依赖裁剪 @2heal1

在 [Module Federation](/guide/features/module-federation) 相关能力上，Rspack 2.0 继续完善共享依赖的裁剪能力。通过 MF share treeshaking，Rspack 可以对 `shared` 依赖做更细粒度的导出级裁剪，减少共享包里实际不会被使用到的部分，从而进一步压缩微前端场景下的依赖体积。

它同时支持 `runtime-infer` 和 `server-calc` 两种模式，分别面向运行时推断和部署侧二次计算这两类不同的工程策略。

例如：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

## 其他更新

除了上面的几项主要能力之外，Rspack 2.0 还补充了一些偏工程细节但很实用的更新。

<span id="import-bytes"></span>
### 静态资源 bytes 导入 @LingyuCoder

Rspack 2.0 还支持将静态资源直接作为 bytes 导入。这样你可以把资源以 `Uint8Array` 的形式读入，再配合 `TextDecoder` 等 API 在运行时进行处理。

```js title="app.js"
import fileBytes from './file.bin' with { type: 'bytes' };

const decoder = new TextDecoder('utf-8');
const text = decoder.decode(fileBytes);
```

<span id="detect-syntax"></span>
### 内置 SWC loader 支持自动检测语法 @chenjiahan

其中一个是 [`builtin:swc-loader` 的 `detectSyntax`](/guide/features/builtin-swc-loader#detectsyntax)。当你用同一条规则同时处理 `.js`、`.jsx`、`.ts`、`.tsx` 等不同类型文件时，`detectSyntax: 'auto'` 可以根据扩展名自动推断 parser 配置，减少重复配置。

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

<span id="automatic-targets"></span>
### 自动继承 targets @ahabhgk

Rspack 2.0 进一步减少了构建配置中重复描述目标环境的成本。[`builtin:swc-loader`](/guide/features/builtin-swc-loader)、[`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader)、[`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin) 和 [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) 的目标环境配置项现在会默认继承顶层的 [`target`](/config/target) 配置，而不再需要开发者分别在 loader、plugin 和顶层目标环境之间手动同步。

这样做可以减少配置冗余，也能避免“转译目标”和“压缩目标”不一致带来的问题。

<span id="hashed-module-ids-plugin"></span>
### HashedModuleIdsPlugin @LingyuCoder

另外，Rspack 2.0 也补充了 `HashedModuleIdsPlugin`。如果你希望更明确地控制模块 id 的生成策略，而不是只依赖默认配置或 `optimization.moduleIds` 的字符串选项，可以直接使用这个插件。

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

<span id="enforce-size-threshold"></span>
### splitChunks.enforceSizeThreshold @jserfeng

此外，Rspack 2.0 也补充了 `splitChunks.enforceSizeThreshold` 选项。它适合在代码分割场景中为 chunk 体积设置一个强制阈值；当模块组大小超过这个阈值时，会跳过部分 `maxRequests` 限制，从而更积极地执行拆分策略。

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      enforceSizeThreshold: 50000,
    },
  },
};
```

<span id="better-tree-shaking"></span>
### 更好的 tree shaking @ahabhgk

Rspack 2.0 的 [tree shaking](/guide/optimization/tree-shaking) 能力在 1.x 的基础上继续增强。在 1.6 中，Rspack 已经增强了动态导入场景下的 tree shaking；而 2.0 又新增了 [`experiments.pureFunctions`](/guide/migration/rspack_1.x#新增-experimentspurefunctions用于函数粒度-tree-shaking)，允许通过 `/*#__NO_SIDE_EFFECTS__*/` 注解或 [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) 配置，对跨模块的 pure function 调用做更细粒度分析。
