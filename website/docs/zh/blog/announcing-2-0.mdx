# Rspack 2.0 发布公告

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

距离 Rspack 1.0 正式发布已经过去了 1 年 8 个月。在这期间，我们保持着每周发布一个小版本、每两个月发布一个中版本的节奏，逐步对齐了 webpack 如持久化缓存、Layers、Lazy Compilation 等等诸多特性，并补齐了大量插件所需的钩子和 Node API，让更多项目能够以更低成本迁移到 Rspack。

同时我们也没有放下对极致性能的追求，在不断优化自身算法和数据结构的同时，也探索并实现了如增量构建、Lazy Barrel、原生文件系统监听器等优化，使 Rspack 成为一个既兼容 webpack 生态、又具备更高构建性能的打包方案。

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

在这个过程中，Rspack 逐渐被越来越多的前端开发者认可。相较于 1.0 发布时，Rspack 的周下载量增长了约 **48 倍**。并且 Docusaurus、Nuxt、Nx、Next.js、Storybook、Meteor、Remotion、Addfox、Vize 等框架、工具和平台也已接入 Rspack。这说明 Rspack 已逐步成长为一个在开源社区中得到广泛验证的 bundler 方案。

![Rspack 2.0 repo stats](https://assets.rspack.rs/rspack/assets/rspack-v2-0-repo-stats.png)

也正是如此，Rspack 收到来自社区源源不断的支持。自 1.0 发布以来，Rspack 仓库累计收到了 **4,728 个 PR** 和 **1,052 个 Issue**，共有 **164 位贡献者**参与共建。感谢所有社区贡献者，正是你们持续的反馈、讨论和代码贡献，推动着 Rspack 不断前进。

## 尝试 2.0

### 新用户安装

如果你是新用户，推荐直接创建一个基于 Rspack 2.0 的项目开始体验。需要注意的是，Rspack 2.0 要求 **Node.js 20.19+** 或 **22.12+**。

如果你希望获得一套开箱即用、默认由 Rspack 驱动的构建体验，我们推荐直接使用 [Rsbuild](https://rsbuild.rs/) 创建项目：

```bash
npm create rsbuild@latest
```

如果你更希望直接使用 [Rspack CLI](/api/cli)，也可以执行：

```bash
npm create rspack@latest
```

然后按照提示完成初始化即可。更多安装方式可以参考 [快速上手](/guide/start/quick-start) 文档。

### 老用户升级

Rspack 2.0 在带来不少新特性的同时，它也移除了诸多历史上已废弃的功能。因此，建议先参考 [迁移指南](/guide/migration/rspack_1.x)，其中已经整理了从 v1 到 v2 的主要不兼容变更和对应的迁移方式。

当然我们更推荐直接借助 Coding Agent 和我们的 [Skills](https://github.com/rstackjs/agent-skills) 来完成迁移。按照如下命令安装 `rspack-v2-upgrade` 后，就可以让 Agent 结合你的项目配置和官方迁移规则完成升级流程。对于配置项较多、历史包袱较重的项目，这会比手工逐项排查更高效。

```
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## 后续方向

Rspack 2.0 不仅是一个里程碑，也意味着一个新的起点。

长期来看，Rspack 团队会将工作重心从“兼容 webpack”逐步转向**构建性能**、**产物体积**、**社区规范**、**Agent 协同**等方向，结合 Rstack 生态中的一整套开发工具，为开发者提供“AI 友好的一体化 JavaScript 工具链”。

而在短期来看，团队将着眼于如下工作：

- 持续改进 tree shaking 和 side effects 分析能力，并探索函数级的模块切分方案，以进一步降低产物体积
- 持续优化构建链路上各个环节的算法和数据结构，并将它们接入到增量构建和持久化缓存当中，以提升各种场景下的构建速度
- 持续增强 ESM 产物的质量和稳定性，并与社区中的其他团队共同探讨 bundler 对 ESM 规范更好的支持方式
- 持续建设 [Rstack Skills](https://github.com/rstackjs/agent-skills)，为开发者提供更多 AI 友好的工具来提升开发效率，降低问题排查、产物分析的成本

## 常见问题

### 2.0 以后，Rspack 是否不再兼容 webpack？

不是。Rspack 后续仍会保持有限的 webpack 兼容性，并尽可能让现有项目能够继续平滑迁移和使用。只是从 2.0 开始，Rspack 不会再把“兼容”作为唯一目标；当我们在构建性能、产物体积、社区规范、Agent 支持等方向上找到了更适合的方案，并且这些方案必须打破一部分历史兼容时，我们会选择做出相应调整。

但即使如此，我们仍会尽量提供详细的过渡方式、升级说明和迁移方案，帮助用户理解这些变化，并以可控的成本完成迁移。

### 2.0 发布后，1.x 版本是否会完全停止维护？

不会。2.0 发布后，Rspack 的主要演进方向会逐步转向 2.x，但这并不意味着 1.x 会在发布后立刻完全停止维护。对于仍在使用 1.x 的项目，我们会根据实际情况在一个较长的时间段内提供必要维护，例如关键问题修复或迁移期支持。

但从长期来看，新增能力和主要优化都会优先在 2.x 上推进，因此如果你的项目还停留在 1.x，我们建议尽早规划升级。

## 更快的构建速度

![Rspack 2.0 performance comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-perf-comparison.png)

在 Rspack 2.0 中，我们移除了部分废弃的功能和代码，升级了部分老旧的依赖，也因此有机会进一步优化内部的算法和数据结构，这也带来了约 **10%** 的性能收益。

<span id="rsc-support"></span>
## 内置 React Server Components 支持 @SyMind

在 Rspack 2.0 中，[React Server Components](/guide/tech/rsc) 将在核心中直接提供了面向 RSC 的插件与编译支持，而不再完全依赖上层框架做额外拼装。

通过 [`rsc.createPlugins()`](/guide/tech/rsc#配置插件) 生成的 `ServerPlugin` 与 `ClientPlugin`，Rspack 可以协调 Server Compiler 和 Client Compiler 两套构建流程，自动收集 `"use client"`、`"use server"`、`"use server-entry"` 等指令信息，生成 RSC 运行时所需的 manifest，并在服务端组件变更时通过 [`onServerComponentChanges`](/guide/tech/rsc#开发服务器) 暴露更新信号。

与此同时，`builtin:swc-loader` 也被增强为可以识别并转换这些 RSC 指令，使 React 19 时代的 RSC、Server Actions、客户端引用和服务端引用能够在同一套构建体系里协同工作。

例如：

```js title="rspack.config.mjs"
import { experiments } from '@rspack/core';

const { createPlugins } = experiments.rsc;
const { ServerPlugin, ClientPlugin } = createPlugins();

export default [
  {
    target: 'web',
    plugins: [new ClientPlugin()],
  },
  {
    target: 'node',
    plugins: [new ServerPlugin()],
  },
];
```

<span id="portable-cache"></span>
## 可移植缓存 @jerrykingxyz

在 Rspack 1.2 中我们引入了[持久化缓存](/config/cache#persistent-cache)，但这份缓存无法在不同机器、不同路径、不同环境之间复用，这使得团队协作、远程构建等场景难以通过缓存降低构建的耗时。

因此，在 Rspack 2.0 中，我们引入了可移植缓存（Portable Cache），你可以通过 [`cache.portable`](/config/cache#cacheportable) 来生成/使用一份可以拷贝的缓存数据：

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

除了可移植缓存之外，Rspack 2.0 也提供了 [`cache.readonly`](/config/cache#cachereadonly) 选项。开启后，缓存只会从磁盘读取而不会写回，这对于 CI 场景尤其有用，可以直接复用预热缓存而不修改它。

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

<span id="pure-esm"></span>
## 纯 ESM 包 @chenjiahan

为了让 Rspack 包形态更符合现代 Node.js / ESM 生态规范，`@rspack/core`、`@rspack/cli`、`@rspack/dev-server` 和 `@rspack/plugin-react-refresh` 等包被正式切换为纯 ESM 包发布，CommonJS 的构建产物也被同步移除。这让模块加载和工具链集成方式更加统一。

因此，你需要将 Node.js 升级到原生支持 `require(esm)` 的 20.19+ 或 22.12+ 版本。

<span id="better-esm"></span>
## 更好的 ESM 产物 @jserfeng

对于库打包场景来说，ESM 产物质量尤为重要。越纯净的 ESM 产物越有利于静态分析，下游的代码分割和 tree shaking 的效果也会越好。

Rspack 2.0 在此前实验性的 `EsmLibraryPlugin` 基础上进一步提升了 ESM 产物质量，同时也将配置方式收敛到了更自然的 [`modern-module`](/config/output#outputlibrarytype) 和 [`output.module`](/config/output#outputmodule) 上。

你可以在库构建中这样配置：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
    library: {
      type: 'modern-module',
    },
  },
};
```

<span id="import-meta-properties"></span>
## 更好的 import.meta 支持 @ahabhgk

在源码侧，Rspack 2.0 也继续完善了 `import.meta` 相关能力。对于未知的 `import.meta` 属性，可以通过 [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) 配置，使其不在编译期被过早静态决定，而是保留到运行时求值。

例如，如果你希望未知的 `import.meta` 属性保留到运行时求值，可以显式配置：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

此外，Rspack 2.0 还补充了对 [`import.meta.main`](/api/runtime-api/module-variables#importmetamain)、[`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename) 和 [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname) 等能力的支持。

<span id="import-defer"></span>
## 更好的延迟导入支持 @ahabhgk

Rspack 在 1.6 首次支持了 `import defer` 语法，而在 2.0 中，这项能力继续向更完整的形态推进。除了最初的 `import defer * as foo from './foo'`，Rspack 后续还补齐了 `import.defer()` 的支持，使“导入模块但延后执行”的语义能够覆盖更多真实用法。这让开发者可以更细致地控制模块执行时机和副作用触发时机。

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

<span id="dev-server"></span>
## 更轻量的开发服务器 @chenjiahan

`@rspack/dev-server` 在 2.0 中不再作为 `@rspack/cli` 的默认依赖，这是因为部分场景并不需要开发服务器。如果你在项目中使用 `rspack dev` 或 `rspack serve` 命令，需要手动安装：

```bash
pnpm add -D @rspack/dev-server
```

同时，为了进一步精简 `@rspack/dev-server`，它将引入 `@rspack/dev-middleware` 来替代 `webpack-dev-middleware`，并用 `connect-next` 替代了 Express v4。

<span id="subpath-alias-import"></span>
## 支持 `#/` 子路径别名导入 @stormslowly

在模块解析方面，Rspack 2.0 支持了 `#/` 形式的子路径别名导入。这样一来，你可以直接复用 `package.json` 里的 `imports` 字段来组织包内路径映射，而不必额外维护一套独立的别名配置。

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

<span id="pnp-manifest"></span>
## 支持 resolve.pnpManifest @stormslowly

在 Yarn PnP 场景下，Rspack 2.0 的 resolver 支持了 `resolve.pnpManifest` 选项，用于显式指定 PnP 清单文件路径。这样在复杂的包管理和 monorepo 环境下，你依然可以细致地控制 PnP 元信息的读取与解析行为，而不必完全依赖自动探测：

```js title="rspack.config.mjs"
import path from 'node:path';

export default {
  resolve: {
    pnp: true,
    pnpManifest: path.resolve(import.meta.dirname, '.pnp.cjs'),
  },
};
```

<span id="mf-runtime"></span>
## 可选的模块联邦运行时 @chenjiahan

`@module-federation/runtime-tools` 在 2.0 中也改成了可选依赖，并扩展了兼容版本范围，以同时接受较新的运行时版本。

如果你使用了模块联邦特性，现在需要手动安装 `@module-federation/runtime-tools`：

```bash
pnpm add @module-federation/runtime-tools
```

<span id="mf-share-treeshaking"></span>
## 模块联邦共享依赖裁剪 @2heal1

Rspack 2.0 还完善了在 [Module Federation](/guide/features/module-federation) 共享依赖的裁剪能力，以对 `shared` 依赖做更细粒度的导出级优化，删除共享包中实际不会被使用到的部分，从而进一步减小微前端场景下的产物体积。

它同时支持 `runtime-infer` 和 `server-calc` 两种模式，分别面向运行时推断和部署侧二次计算这两类不同的工程策略。

例如：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

## 其他特性更新

<span id="detect-syntax"></span>
### 内置 SWC loader 自动检测语法 @chenjiahan

当你用同一条规则同时处理 `.js`、`.jsx`、`.ts`、`.tsx` 等不同类型文件时，[`builtin:swc-loader` 的 `detectSyntax`](/guide/features/builtin-swc-loader#detectsyntax) 可以根据扩展名自动推断 parser 配置，以此来避免重复配置。

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

<span id="automatic-targets"></span>
### 自动继承 targets 配置 @ahabhgk

[`builtin:swc-loader`](/guide/features/builtin-swc-loader)、[`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader)、[`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin) 和 [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) 的目标环境配置项现在会默认继承顶层的 [`target`](/config/target) 配置，而不再需要开发者分别在 loader、plugin 和顶层目标环境之间手动同步，减少配置冗余的同时也能避免“转译目标”和“压缩目标”不一致带来的问题。

<span id="hashed-module-ids-plugin"></span>
### 支持 HashedModuleIdsPlugin

`HashedModuleIdsPlugin` 也在 2.0 中得到支持。如果你希望基于模块的相对路径计算出一个简短、稳定且唯一的 id，可以使用这个插件：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

<span id="enforce-size-threshold"></span>
### 支持 splitChunks.enforceSizeThreshold @jserfeng

代码分割选项 `splitChunks.enforceSizeThreshold` 也在 2.0 中得到支持。它可以在代码分割时为 chunk 体积设置一个强制阈值，当模块组大小超过这个阈值时，会跳过部分 `maxRequests` 限制，从而更积极地执行拆分策略。

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      enforceSizeThreshold: 50000,
    },
  },
};
```

<span id="better-tree-shaking"></span>
### 更好的 tree shaking @ahabhgk

2.0 中新增了 `experiments.pureFunctions`，允许通过 `/*#__NO_SIDE_EFFECTS__*/` 注解或 [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) 配置，对跨模块的纯函数调用做更细粒度的分析。

同时一部分过去没有被识别的 CommonJS 导入导出在 2.0 中能也够被正确解析，使得 [tree shaking](/guide/optimization/tree-shaking) 能力进一步增强。

## 其他变更

- [`module.parser.javascript.exportsPresence`](/config/module#moduleparserjavascriptexportspresence) 的默认值改为 `error`，缺失导出现在会默认报错而非警告。
- `module.parser.javascript.strictExportPresence` 已被移除，缺失导出的行为统一由 [`module.parser.javascript.exportsPresence`](/config/module#moduleparserjavascriptexportspresence) 控制。
- [`module.parser.javascript.requireAlias`](/config/module#moduleparserjavascriptrequirealias) 的默认值改为 `false`，改名后的 `require` 调用将不再被解析。
- [`builtin:swc-loader`](/guide/features/builtin-swc-loader) 不再读取 `.swcrc`，SWC 配置需迁移到 loader 选项中。
- [`builtin:swc-loader`](/guide/features/builtin-swc-loader) 的 `rspackExperiments.collectTypeScriptInfo` 配置已被移除，TypeScript 信息收集现在统一通过 [`collectTypeScriptInfo`](/guide/features/builtin-swc-loader#collecttypescriptinfo) 配置。
- `builtin:swc-loader.rspackExperiments.import` 已迁移到顶层 [`transformImport`](/guide/features/builtin-swc-loader#transformimport)。
- `output.libraryTarget`、`output.libraryExport`、`output.umdNamedDefine` 和 `output.auxiliaryComment` 都已迁移到 [`output.library.*`](/config/output#outputlibrary) 下。
- [`output.chunkLoadingGlobal`](/config/output#outputchunkloadingglobal) 的默认值改为 `rspackChunk${output.uniqueName}`。
- [`output.hotUpdateGlobal`](/config/output#outputhotupdateglobal) 的默认值改为 `rspackHotUpdate${output.uniqueName}`。
- [`output.bundlerInfo.force`](/config/output#outputbundlerinfo) 的默认值改为 `false`，bundler 信息不再强制注入。
- [`output.trustedTypes.policyName`](/config/output#outputtrustedtypes) 在没有设置 `output.uniqueName` 时的兜底值从 `'webpack'` 改为 `'rspack'`。
- `output.charset` 已被移除，现代浏览器会按 UTF-8 处理脚本。
- `optimization.removeAvailableModules` 已被移除。
- [`stats.toJson()`](/api/javascript-api/stats#tojson) 默认不再返回 `modules`、`assets`、`chunks`、`chunkGroups` 和 `entryPoints`。
- `profile` 和 `stats.profile` 已被移除，建议使用 [Rsdoctor](/guide/optimization/use-rsdoctor) 分析性能。
- [`WarnCaseSensitiveModulesPlugin`](/plugins/webpack/case-sensitive-plugin) 已被 [`CaseSensitivePlugin`](/plugins/webpack/case-sensitive-plugin) 替代。
- [`HtmlRspackPlugin`](/plugins/rspack/html-rspack-plugin) 和 [`RuntimePlugin`](/api/plugin-api/runtime-plugin-hooks) 上的 `getHooks` 别名已被移除，请使用 `getCompilationHooks` 替代。
- [`HtmlRspackPlugin`](/plugins/rspack/html-rspack-plugin) 和 [`RuntimePlugin`](/api/plugin-api/runtime-plugin-hooks) 的 `sri` 配置已被移除，SRI 相关能力现在请使用 [`SubresourceIntegrityPlugin`](/plugins/rspack/subresource-integrity-plugin)。
- [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin.mdx) 的 `draft` 配置已被移除并由 `drafts` 取代，同时 `cssHeadDataCompression` 已被移除。
- JavaScript 请求默认的 [`resolve.extensions`](/config/resolve#resolveextensions) 不再包含 `.wasm`。
- CSS `@import` 默认的解析 [`conditionNames`](/config/resolve#resolveconditionnames) 不再包含 `webpack`。
- [`devtool`](/config/devtool) 在开发模式下的默认值改为 `cheap-module-source-map`，在 CLI 生产模式下的默认值改为 `false`。
- [`resolve.roots`](/config/resolve#resolveroots) 的默认值从 `[context]` 改为 `[]`。
- Runtime module 上废弃的 `constructorName`、`moduleIdentifier` 等属性已被移除。
- [`ProgressPlugin`](/plugins/webpack/progress-plugin) 自定义 handler 的第三个参数从 `items: string[]` 改成了结构化的 `info` 对象。
- `NormalModule.getCompilationHooks(compilation).readResourceForScheme` 已被移除，改用 `readResource.for(scheme)`。
- `module.unsafeCache` 已被移除。
- `experiments.css` 已被移除，[CSS](/guide/tech/css) 能力现在默认可用。
- [`experiments.asyncWebAssembly`](/config/experiments#experimentsasyncwebassembly) 的默认值改为 `true`。
- `experiments.cache` 已迁移到顶层 [`cache`](/config/cache)。
- `experiments.rspackFuture` 已被移除，`bundlerInfo` 迁移至 [`output.bundlerInfo`](/config/output#outputbundlerinfo)。
- `experiments.incremental` 已迁移到顶层 [`incremental`](/config/incremental)。
- `experiments.layers` 已被移除，[layers](/guide/features/layer) 能力默认可用。
- `experiments.topLevelAwait` 已被移除，top-level await 解析能力默认可用。
- `experiments.lazyCompilation` 已迁移到顶层 [`lazyCompilation`](/config/lazy-compilation)。
- `experiments.lazyCompilationMiddleware` 已迁移到顶层 [`lazyCompilationMiddleware`](/guide/features/lazy-compilation#与自定义的服务器集成)。
- `experiments.lazyBarrel` 已被移除，[lazy barrel](/guide/optimization/lazy-barrel) 优化默认开启。
- `experiments.typeReexportsPresence` 已被移除，其行为现通过 [`module.parser.javascript.typeReexportsPresence`](/config/module#moduleparserjavascripttypereexportspresence) 控制。
- `experiments.inlineConst` 已被移除，常量内联现通过 [`optimization.inlineExports`](/config/optimization#optimizationinlineexports) 控制。
- `experiments.inlineEnum` 已被移除，枚举内联现通过 [`optimization.inlineExports`](/config/optimization#optimizationinlineexports) 配合 [`collectTypeScriptInfo.exportedEnum`](/guide/features/builtin-swc-loader#collecttypescriptinfoexportedenum) 控制。
- `experiments.outputModule` 已被移除，[ESM 格式产物](/guide/features/esm)输出现通过 [`output.module`](/config/output#outputmodule) 控制。
- `experiments.parallelLoader` 已被移除，[并行 loader](/config/module-rules#rulesuseparallel) 能力默认可用。
