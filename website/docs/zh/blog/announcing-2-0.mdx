# Rspack 2.0 发布公告

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

距离 Rspack 1.0 正式发布，已经过去了 1 年 8 个月。

2024 年 8 月发布 1.0 时，我们给 Rspack 设定了一个非常明确的阶段目标：在尽可能保持 webpack API 和生态兼容的前提下，帮助项目以可控的迁移成本获得 5 到 10 倍的构建性能提升。

从今天回看，1.x 基本完成了这个阶段的任务。Rspack 逐步补齐了持久化缓存、Layers、Lazy Compilation 等关键能力，也补全了大量插件依赖的钩子和 Node API，让更多项目可以在不重写工程体系的前提下迁移到 Rspack。

与此同时，我们也没有把 1.x 做成单纯的“webpack 替代品”。在持续对齐生态的同时，Rspack 也引入了增量构建、Lazy Barrel、原生文件系统监听器等能力，让兼容性和性能能够在同一套工具中成立。

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

在这个过程中，Rspack 也在越来越多的真实项目中得到验证。与 1.0 发布时相比，Rspack 的周下载量增长了约 **48 倍**。Docusaurus、Nuxt、Nx、Next.js、Storybook、Meteor、Remotion、Addfox、Vize 等框架、工具和平台都已经接入 Rspack。

这说明 Rspack 已经从一个主要面向迁移场景的选择，逐步成为社区里可以稳定使用并持续演进的 bundler 方案。

社区的支持也在持续增长。自 1.0 发布以来，Rspack 仓库累计收到了 **4,728 个 PR** 和 **1,052 个 Issue**，共有 **164 位贡献者**参与共建。

感谢所有社区贡献者。正是这些持续的反馈、讨论和代码贡献，推动着 Rspack 不断向前。

与此同时，前端社区也在继续变化。Node.js 对 ESM 的支持更加成熟，新的模块规范和运行时能力在不断演进，AI 也在显著降低工具迁移和问题排查的成本。随着这些变化逐步成为日常开发的一部分，我们也开始重新思考：一个面向未来的 bundler，应该默认提供怎样的能力。

正是在这样的背景下，我们在 2025 年启动了 Rspack 2.0 的设计与迭代。经过数十个 beta 和 RC 版本的打磨之后，Rspack 2.0 现在正式发布。

## 更加面向未来的设计 \{#future-oriented-design}

与 1.0 主要聚焦于兼顾既有生态不同，2.0 更强调在保持迁移成本可控的前提下，为未来的 JavaScript 工程提供更合适的默认设计。

在 1.x 阶段，Rspack 有意让诸多 API 和默认值尽量与 webpack 5 对齐。这一策略有效降低了迁移成本，也使更多上层工具和框架能够较早接入 Rspack。随着 ESM、`import.meta`、`import defer` 等现代模块能力逐步走向主流，一部分历史设计已不再是当前语境下的最优选择。对于 bundler 而言，需要考虑的不仅是对既有行为的兼容，还包括更合理的默认值、更清晰的配置模型，以及更有利于性能和产物优化的底层抽象。

基于这一判断，Rspack 2.0 对部分历史设计进行了重新审视。Rspack 仍将保持对 webpack 生态的高兼容性；在此基础上，我们也会优先采用更符合现代工程需求的 API、默认值与产物形态，并通过迁移指南与升级工具将迁移成本控制在合理范围内。

## 内置 React Server Components 支持 \{#rsc-support}

React Server Components（简称 RSC）是 React 提供的一种组件模型：它允许一部分组件直接在服务端执行和渲染，再把结果传递给客户端；与此同时，需要交互的部分仍然可以作为 Client Components 在浏览器中运行。

这种模型可以让数据获取更贴近服务端，减少发送到客户端的 JavaScript 体积，并更自然地把“服务端渲染内容”和“客户端交互能力”组合在同一个应用里。因此，RSC 也正在成为 React 全栈框架中越来越重要的一类基础能力。

Rspack 2.0 内置了对 React Server Components 通用的底层构建能力的支持，包括：

1. 编译期检查：在编译阶段拦截违反 RSC 规范的 React API 调用，提前规避运行时错误。
2. CSS 支持：在编译阶段收集服务端与客户端组件样式，并在渲染时注入。
3. HMR 支持：同时支持服务端和客户端组件的热更新。

你可以通过如下两种方式来使用它：

- **使用 Rsbuild**：Rsbuild 提供对 React Server Components 的开箱即用支持，可以快速创建一个 RSC 项目。详见 [rsbuild-plugin-rsc](https://github.com/rstackjs/rsbuild-plugin-rsc)。
- **手动配置 Rspack**：你也可以参考 [React Server Components 文档](https://v2.rspack.rs/guide/tech/rsc)，手动添加 React Server Components 相关配置。

[rspack-rsc-examples](https://github.com/rstackjs/rstack-examples/tree/main/rspack/rspack-rsc) 仓库中提供了使用 Rspack 构建 RSC 应用的完整示例。

生态落地与合作：

- Modern.js 框架目前已采用 Rspack 的 RSC 实现（[详见 Modern.js RSC 文档](https://modernjs.dev/guides/basic-features/render/rsc.html)）。
- 支持 React Router Data Mode（[React Router 示例](https://github.com/rstackjs/rsbuild-plugin-rsc/tree/main/examples/react-router)）。
- 我们正与 [TanStack](https://tanstack.com/) 展开深度合作，全面支持其利用 Rspack 作为底层引擎来编译和构建 RSC 应用。

## 更快的构建速度 \{#build-performance}

![Rspack 2.0 performance comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-perf-comparison.png)

在 Rspack 2.0 中，我们继续整理构建主流程。我们升级了一批老旧依赖，清理了一些已经废弃的代码路径，也优化了性能热点上的算法和数据结构。

以 [10K 个 React 组件 benchmark](https://github.com/LingyuCoder/rspack-react-10k-benchmark/actions/runs/24661521793) 为例，2.0 在生产构建和 HMR 两个流程上都实现了约 **10%** 的性能提升。

而针对开启了[持久化缓存](/config/cache#persistent-cache)场景，我们也在缓存的流程覆盖和内存消耗上更进一步：

- 将 [`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin) 接入到了持久化缓存，代码压缩结果可以跨构建复用，使得大型项目的热启动性能提升约 **80%**。
- 优化持久化缓存的底层存储实现，使大型项目开启持久化缓存时的内存占用降低了 **20%+**。

## 更少的默认依赖 \{#fewer-deps}

Rspack 2.0 减少了默认安装的 npm 依赖数量：

- `@rspack/dev-server` 的依赖数量从 192 降至 1，安装体积减少超过 90%，从 15 MB 降至 1.4 MB。
- `@rspack/core` 的依赖数量从 8 降至 1。
- `@rspack/cli` 现在为零依赖。

在 Rspack v1 中，`@rspack/dev-server` 通过 `webpack-dev-server` 间接引入了较多依赖，这增加了安装体积，也提高了依赖管理的复杂度。为此，我们对 `@rspack/dev-server` 进行了重构，重新梳理并精简了功能和默认依赖。

同时，`@rspack/cli` 也不再默认依赖 `@rspack/dev-server`，这意味着如果只使用构建能力，将不再需要引入开发服务器相关的依赖。

此外，我们还采用了以下方式来减少依赖：

- 将非核心依赖调整为可选依赖。例如 [@module-federation/runtime-tools](https://www.npmjs.com/package/@module-federation/runtime-tools) 现在仅在使用 [模块联邦插件](/plugins/webpack/module-federation-plugin) 时才需要手动安装。
- 使用更轻量的替代实现，例如以 [connect-next](https://github.com/rstackjs/connect-next) 替代 Express。对于开发服务器而言，Connect 的中间件模型已经能够覆盖大多数场景，这也与 Rsbuild 的开发服务器实现保持一致。
- 将部分依赖 [打包进发布产物](https://e18e.dev/blog/bundling-dependencies)，由 Rspack 统一控制这些依赖及其子依赖的版本，避免依赖自动升级带来的潜在供应链风险。
- 优先使用 Node.js 20+ 内置的原生 API，例如用 [styleText](https://nodejs.org/api/util.html#utilstyletextformat-text-options) 替代 `picocolors`。

## 改进 tree shaking \{#better-tree-shaking}

Rspack 2.0 在 tree shaking 上继续补齐了若干过去难以静态分析的代码场景。现在，以下几类写法也能够被更准确地识别并参与 [tree shaking](/guide/optimization/tree-shaking)：

- **CommonJS `require` 解构赋值**：例如 `const { a } = require("./m")`，Rspack 现在可以识别其中实际使用到的导出。
- **CommonJS `require` 变量成员访问 / 调用**：例如 `const m = require("./m"); m.a; m.b();`，Rspack 现在也能继续向下分析具体使用到的成员。
- **动态 `import()` 结果的内联成员访问 / 调用**：例如 `(await import("./m")).a` 以及类似的成员访问、成员调用形式，也能够参与导出级别的裁剪。

此外，2.0 还新增了 `experiments.pureFunctions`，用于对 pure function 调用做更细粒度的 tree shaking。你既可以在源码中通过 `/*#__NO_SIDE_EFFECTS__*/` 注解标记函数定义，也可以通过 [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) 为无法直接修改源码的第三方模块手动标记顶层导出函数。

例如：

```js title="rspack.config.mjs"
export default {
  mode: 'production',
  experiments: {
    pureFunctions: true,
  },
};
```

```js title="utils.js"
/*#__NO_SIDE_EFFECTS__*/
export function joinClassName(a, b) {
  return `${a}-${b}`;
}
```

```js title="index.js"
import { joinClassName } from './utils';

joinClassName('btn', 'primary');
console.log('hello');
```

在这个例子里，`joinClassName` 被标记为“调用无副作用”，当它的返回值没有被使用时，这次调用就可以在构建产物中被安全移除。

## 模块联邦共享依赖 tree shaking \{#mf-share-treeshaking}

除了对普通模块导入导出的分析，Rspack 2.0 也继续完善了[模块联邦](/guide/features/module-federation) 场景下的共享依赖 tree shaking。它可以对 `shared` 依赖做导出级别的优化，移除实际不会被使用到的部分，从而减少微前端场景中的共享包体积。

在 Rspack 1.x 的模块联邦共享依赖方案里，只要一个依赖被声明到 `shared`，运行时通常就需要准备完整的依赖包。即使应用实际只用到了其中少量导出，也往往会把整份依赖带进来。对于 `antd`、`lodash-es` 这类体积较大、导出较多的库来说，这会明显增加共享包体积。

在 Rspack 2.0 中，当某个 `shared` 项开启 `treeShaking` 后，`ModuleFederationPlugin` 会为这类共享依赖生成对应的 tree shaking 产物。运行时会优先加载这份更小的结果；如果当前请求无法命中，再回退到完整依赖，以保证行为稳定。

它同时支持 `runtime-infer` 和 `server-calc` 两种模式：

- `runtime-infer` 适合先快速接入和本地验证。运行时会优先复用已经加载过的 tree shaking 共享包；如果当前需要的导出无法满足，则自动回退到完整依赖。你也可以通过 `usedExports` 预先声明常用导出，以提升命中率。
- `server-calc` 更适合多应用协作的线上场景。你可以在部署侧统一汇总各个消费者真正会用到的导出，再生成更稳定的共享依赖 tree shaking 结果。

对于组件库这类典型场景，这项能力可以把“整包共享”变成“按实际使用结果共享”，减少不必要的代码加载。更完整的配置与工作方式，可以参考 [共享依赖 tree shaking 指南](https://module-federation.io/zh/guide/performance/shared-tree-shaking)。

例如：

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

## 纯 ESM 包 \{#pure-esm}

Rspack 的核心包现在以 pure ESM 包的形式发布，并移除了自身的 CommonJS 构建产物，这使模块加载方式更加统一，也更符合当前 Node.js 的主流实践。

本次变更涉及以下 npm 包：

- [@rspack/core](https://www.npmjs.com/package/@rspack/core)
- [@rspack/cli](https://www.npmjs.com/package/@rspack/cli)
- [@rspack/dev-server](https://www.npmjs.com/package/@rspack/dev-server)
- [@rspack/dev-middleware](https://www.npmjs.com/package/@rspack/dev-middleware)
- [@rspack/plugin-react-refresh](https://www.npmjs.com/package/@rspack/plugin-react-refresh)

在 Node.js 20 及以上版本中，运行时已原生支持通过 [require(esm)](https://nodejs.org/api/modules.html#loading-ecmascript-modules-using-require) 加载 ESM 模块。因此，对大多数仍通过 JavaScript API 使用 Rspack 的项目来说，这一变更通常不会带来实际影响，也无需额外修改现有代码。

## 改进 ESM 产物 \{#better-esm}

对于库构建场景，ESM 产物的质量非常重要。产物越纯净，越有利于静态分析，也越有利于下游继续做代码分割和 tree shaking。

在 Rspack 1.6 中，我们引入了实验性的 [`rspack.experiments.EsmLibraryPlugin`](https://v1.rspack.rs/plugins/rspack/esm-library-plugin)，用一条专用的渲染路径替代此前基于 module concatenation 的 ESM 输出方案。之后的若干个小版本里，这条路径持续被打磨，并逐步接管了产物流水线中更多的环节。到 2.0，它已经完全稳定。

与此同时，我们移除了实验性插件入口。现在你只需要将 [`output.library.type`](/config/output#outputlibrarytype) 配置为 `'modern-module'`，就可以启用这条路径：

```js title="rspack.config.mjs"
export default {
  output: {
    library: {
      type: 'modern-module',
    },
  },
};
```

与通用的 [`output.module`](/config/output#outputmodule) 不同，`modern-module` 会完整接管库产物的渲染流程。前者仍沿用常规的 chunk / runtime 渲染流程，再将入口导出转换为 `export` 语句；而 `modern-module` 会直接走一条更适合库产物的专用路径，具体包括：

- 使用独立的 scope hoisting 算法，而不再依赖 [`optimization.concatenateModules`](/config/optimization#optimizationconcatenatemodules)，可以在不引入 Rspack 运行时包装代码的前提下，将多个模块合并到同一个 ESM 产物中。
- 直接将 chunk 渲染为 ESM 文件，并使用真实的 `import` / `export` 语句在 chunk 之间进行链接，即使经过[代码分割](/guide/optimization/code-splitting)，产物依然能被下游工具静态分析。
- 自动对会在多个 chunk 中重复出现的模块进行去重，以减少库被拆分为多个入口时的冗余代码。
- 启用 [`output.library.preserveModules`](/config/output#outputlibrarypreservemodules) 能力：在产物中保留源码目录结构，适合需要“源码一个文件、产物一个文件”发布形态的库。

在配置层面，`modern-module` 也内置了一组默认值，用来简化手动配置：它会自动将 [`output.module`](/config/output#outputmodule) 置为 `true`，将 `output.chunkLoading` 设为 `'import'`，并放宽 [`splitChunks`](/plugins/webpack/split-chunks-plugin) 的默认值，以保证拆分后产物的纯净度。更完整的 ESM 场景和配置说明可参考 [ESM 指南](/guide/features/esm)。

## 改进 import.meta 支持 \{#import-meta-properties}

Rspack 2.0 继续完善了 `import.meta` 相关能力。

对于编译阶段无法静态确定的 `import.meta` 属性，你可以通过 [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) 控制其行为，让它不要在编译阶段被转译，而是保留到运行时再求值。

例如：

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

此外，Rspack 2.0 还补充了对 [`import.meta.main`](/api/runtime-api/module-variables#importmetamain)、[`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename) 和 [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname) 等值的支持。

## 改进延迟导入支持 \{#import-defer}

Rspack 在 1.6 在开启 deferImport 时能够解析 `import defer * as foo from './foo'` 语法，而在 2.0 中进一步支持了 `import.defer()`，使“导入模块但延后执行”的语义能够覆盖更多真实用法。

这允许你更细致地控制模块执行时机和副作用触发时机。

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
const file = Math.random() > 0.5 ? 'a.js' : 'b.js';
import.defer('./dir' + file);
```

## 其他特性更新

### 支持 `#/` 子路径别名导入 @stormslowly \{#subpath-alias-import}

在模块解析方面，Rspack 2.0 支持了 `#/` 形式的子路径别名导入。这样你就可以直接复用 `package.json` 中的 `imports` 字段来组织包内路径映射，而不必额外维护一套独立的别名配置。

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

### 简化目标环境配置 \{#automatic-targets}

在 Rspack 1.x 中，顶层的 [target](/config/target) 选项不会影响 loader 或压缩工具的转换目标，你通常还需要在 loaders 或压缩插件的选项中分别配置目标环境选项。这带来了额外的配置成本，同一份目标环境信息需要在多个位置重复声明。

Rspack 2.0 对此进行了改进，内置的 loaders 和压缩插件会默认解析顶层的 `target` 配置，并据此推断各自需要的目标环境设置。多数情况下，你只需要在顶层声明一次目标环境，就能让 JavaScript 代码转换、CSS 转换和代码压缩保持目标一致。

```js title="rspack.config.mjs"
export default {
  target: 'browserslist: Chrome >= 100',
  module: {
    rules: [
      {
        test: /\.js$/,
        use: [
          {
            loader: 'builtin:swc-loader',
            options: {
              env: { targets: 'Chrome >= 100' }, // [!code --]
            },
          },
        ],
      },
    ],
  },
  optimization: {
    minimizer: [
      new rspack.SwcJsMinimizerRspackPlugin(),
      new rspack.LightningCssMinimizerRspackPlugin({
        minimizerOptions: { targets: 'Chrome >= 100' }, // [!code --]
      }),
    ],
  },
};
```

该变更影响以下 loaders 和插件：

- [swc-loader](/guide/features/builtin-swc-loader)
- [lightningcss-loader](/guide/features/builtin-lightningcss-loader)
- [SwcJsMinimizerRspackPlugin](/plugins/rspack/swc-js-minimizer-rspack-plugin)
- [LightningCssMinimizerRspackPlugin](/plugins/rspack/lightning-css-minimizer-rspack-plugin)

### 简化 swc-loader 配置 \{#detect-syntax}

在以往的配置中，如果希望 [builtin:swc-loader](/guide/features/builtin-swc-loader) 同时处理 `.js`、`.jsx`、`.ts`、`.tsx` 等文件，需要根据文件类型显式指定 `jsc.parser` 中 的 `syntax`、`jsx`、`tsx`等选项，以确保 SWC 能以正确的语义解析代码，例如将 `<` 解析为 JSX 语法还是 TypeScript 泛型。

这带来了一定的配置复杂度。为了让 SWC 正确解析不同后缀的文件，通常需要定义多条 `rules`，导致配置变得冗长，不利于维护。

为了解决这一问题，Rspack 2.0 在内置的 `swc-loader` 中引入了 [detectSyntax](/guide/features/builtin-swc-loader#detectsyntax) 选项。启用 `detectSyntax: 'auto'` 后，loader 会根据文件扩展名自动推断 `syntax`、`jsx`、`tsx` 等 `parser` 选项。通过一条规则即可覆盖不同的文件类型，使配置更清晰。

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

### 支持 CSS parser 的 resolveImport 选项 \{#resolve-import}

Rspack 2.0 在 CSS parser 上提供了 [`resolveImport`](/config/module#moduleparsercssautoresolveimport) 选项，`css`、`css/auto`、`css/module` 均可使用。它用于控制构建期是否解析并内联 `@import` at-rule。

当值为 `true`（默认值）时，`@import` 会被视作依赖处理，并用被导入文件的内容替换掉；设置为 `false` 时，`@import` 语句会原样保留在产物中，交由浏览器或下游工具解析。

该选项也可以接收一个函数。Rspack 会对每条 `@import` 规则调用一次，并传入 URL、发起导入的文件路径，以及该 at-rule 上的完整条件（`media`、`supports`、`layer`）。借此你可以按需决定哪些 import 需要内联。例如，可以将本地样式表内联，而把远端 URL 保留为真实的 `@import` 由浏览器加载：

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      'css/auto': {
        resolveImport: ({ url }) => !/^https?:\/\//.test(url),
      },
    },
  },
};
```

### 支持使用哈希生成模块 ID \{#hashed-module-ids-plugin}

Rspack 2.0 为 `optimization.moduleIds` 新增了 `hashed` 选项。启用后，Rspack 会基于模块的相对路径生成哈希值，并将其用作模块 ID。

这适合希望模块 ID 更短、同时又保持稳定性的场景：

```js title="rspack.config.mjs"
export default {
  optimization: {
    moduleIds: 'hashed',
  },
};
```

### 支持代码分割的强制体积阈值 \{#enforce-size-threshold}

代码分割选项 [`splitChunks.enforceSizeThreshold`](/plugins/webpack/split-chunks-plugin) 在 2.0 中得到了支持。它用于设置一个强制拆分的体积阈值：当某个模块组的体积超过该阈值时，[`splitChunks.maxAsyncRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxasyncrequests)、[`splitChunks.maxInitialRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxinitialrequests) 和 `splitChunks.minRemainingSize` 等限制会被忽略，从而确保拆分仍然发生。

如果没有这个配置，请求数限制可能会阻止一个很大的 cache group 被拆分出来，即使拆分本身是更合理的；而加入该阈值后，一旦模块组体积超过阈值，Rspack 就会跳过这些限制并继续完成拆分。

与 webpack 对齐，Rspack 中 `enforceSizeThreshold` 的默认值在生产模式下为 `50000`，其他模式下为 `30000`。同时，它也支持以对象形式针对不同模块类型（如 JavaScript、CSS）分别设置阈值：

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      // 使用数字形式会对所有模块类型生效
      enforceSizeThreshold: 50000,
      // 也可以按模块类型分别配置
      // enforceSizeThreshold: { javascript: 50000, css: 30000 },
    },
  },
};
```

该选项同样可以在 cache group 粒度上配置。你可以保留较严格的全局请求数限制，同时允许某个 cache group（如 `defaultVendors`）在体积超过指定阈值后更积极地完成拆分。

此外，[`minSize`](/plugins/webpack/split-chunks-plugin#splitchunksminsize)、[`maxSize`](/plugins/webpack/split-chunks-plugin#splitchunksmaxsize) 和 [`minChunks`](/plugins/webpack/split-chunks-plugin#splitchunksminchunks) 等与体积和共享度相关的约束仍会继续生效。

## 后续方向

Rspack 2.0 是一个里程碑，也意味着新的开始。

1.x 主要回答的是“如何让 webpack 项目低成本迁移到高性能 bundler 上”；从 2.x 开始，我们会更多回答另一个问题：面向现代化 Web 开发的 bundler 该如何设计。

### 更优的产物优化

Rspack 和 webpack 一样，在很多默认优化上都会优先保证产物行为的稳定性，尤其是在 side effects 判断等环节通常会更保守。这种策略适合大多数项目，但我们也注意到，对于追求极致产物体积的场景来说，仍然有进一步挖掘的空间。

#### 更高级的 Tree Shaking 优化

Rspack 2.0 已经支持基于 pure annotation 的更细粒度优化。接下来，我们会继续提供更细粒度的 Tree Shaking 配置，让项目可以在“安全默认值”和“更激进的产物优化”之间做出更明确的选择，在保障产物稳定性的同时，更好地满足对体积有更高要求的场景。

与此同时，我们也在探索结合静态类型信息来增强优化分析，借鉴 Prepack、Closure Compiler 和 React Compiler 在静态编译上的一些思路，把更多来自业务代码的可优化信息引入 bundler。

#### 更细粒度的代码拆分架构

今天的 code splitting 最小粒度通常仍然是模块。当 chunk A 和 chunk B 只共享某个模块中的一个很小函数时，运行时仍然可能需要把整个模块都带上。这种粒度在很多项目里已经足够，但对于大型应用和框架场景来说，仍然会带来额外的冗余加载。

未来我们会参考 esbuild、Turbopack 等工具在 module splitting 方向上的经验，探索按照实际复用关系把公共模块继续拆成更小的子模块。在保证缓存稳定性、调试体验和运行时行为可控的前提下，进一步减少 chunk 中不必要的加载。同时，我们也会结合 `import defer` 等更新的模块能力，探索更适合现代应用的加载策略。

### 更优的构建性能

Rspack 会继续优化自身的算法、数据结构以及增量构建路径，但我们也越来越明确地看到：在很多真实项目里，瓶颈并不总是出在 bundler 主流程本身，而更常见地出现在 Tailwind、lint、typecheck 等社区插件和配套工具上。

因此，后续的性能工作不会只关注 Rspack 内核。我们也会继续推动 Rstack 生态中的配套工具演进，例如在 Rslint 里探索基于 tsgo 的 lint 和 typecheck 方案，作为当前常见的 `eslint-webpack-plugin` 和 `fork-ts-checker-webpack-plugin` 组合的替代选择，让整个构建链路都能获得更稳定的性能收益。

### 更好的 AI 集成

越来越多的项目已经开始使用 AI 工具处理 Rspack 相关问题，包括构建性能分析、产物优化和自动化迁移。我们已经通过 [Rstack Skills](https://github.com/rstackjs/agent-skills) 为这类工作提供了一定支持。同时，从实际使用中我们也越来越清楚：AI 是否真正能够帮助解决构建问题，很大程度上取决于三个前提，即是否能够获得足够准确的上下文、是否能够稳定复现问题，以及是否能够以足够低的成本快速验证假设。

接下来，这部分工作会主要围绕四个方向展开：

- 更好的 Debugger 支持，让 AI Agent 能够更容易基于 Rspack 的调试版本排查问题，并把编译期与运行时的信息关联起来。
- 更丰富且更结构化的诊断信息，在编译阶段暴露更多可机器消费的过程数据，例如 Tree Shaking、scope hoisting 等优化阶段的 bailout reason，以及更清晰的模块、chunk 和插件链路信息，帮助 AI 自动分析并迭代优化。
- 更稳定的上下文采集与问题复现能力，例如更清晰地暴露配置解析结果、环境信息和关键编译路径，让 AI 生成的建议更容易被复现和验证。
- 更快的反馈循环，为 Agent 场景探索更高效的编译模式，缩短“提出假设、执行验证、继续迭代”的闭环时间。

### 更现代化的产物格式

在 1.x 和 2.0 阶段，Rspack 一直兼容 webpack 的产物格式。这对迁移正确性和生态兼容非常重要，但也意味着我们需要继承一部分历史包袱。为了同时支持 CJS、UMD、ESM 等不同输出格式，以及 HMR、Module Federation 等运行时能力，传统 bundler 产物通常会在可读性和二次静态分析能力上做出妥协。

未来我们会在保留关键运行时能力的前提下，继续探索更现代的产物形态，例如更纯净的 ESM 输出，以及基于 import maps、`import defer`、module fragments 等新标准的组织方式，让产物既更适合运行，也更适合被下游工具继续分析和优化。

### 持续跟进社区规范与平台能力

JavaScript 社区仍在持续演进，新的语法、模块能力和运行时接口也在不断成熟。Rspack 会继续第一时间跟进这些变化，把对新规范的支持尽可能早地带给上层工具和框架。2.0 中已经可以看到这一方向，例如对 pure ESM、`import.meta` 和 `import defer` 的持续完善；后续我们会继续沿着这条路线推进。

### 更丰富的能力支持

今天的 bundler 已经不只是把模块打成 bundle 的工具。对于框架和上层工具来说，它还需要承担多环境编译、服务端与客户端边界分析、样式与资源收集、运行时协调等更底层的职责。

Rspack 2.0 已经在 React Server Components 上迈出了一步。接下来，我们会继续把更多能力沉淀为稳定的框架原语，而不是把复杂度全部留给框架层自行实现。这样，上层框架和工具在接入 Rspack 时，可以用更少的适配代码获得更完整的默认能力。

## 常见问题 \{#faq}

### 2.0 以后，Rspack 是否不再兼容 webpack？

不是。Rspack 仍会尽量保持与 webpack 的兼容性，持续帮助现有项目平滑迁移和使用。

变化在于，从 2.0 开始，Rspack 会在继续重视兼容性的同时，更积极地推进构建性能、产物体积、社区规范和 Agent 支持等方向的演进。当更合适的方案需要对部分历史行为做出调整时，我们会在充分评估影响后稳步推进。

即便如此，我们仍会尽量提供清晰的过渡方式、升级说明和迁移方案，帮助用户理解这些变化，并以可控的成本完成迁移。

### 2.0 发布后，1.x 版本是否会完全停止维护？

不会。2.0 发布后，Rspack 的主要演进方向会逐步转向 2.x，但这并不意味着 1.x 会立刻完全停止维护。对于仍在使用 1.x 的项目，我们会根据实际情况，在一段较长的时间内继续提供必要的维护，例如关键问题修复或迁移期支持。

但从长期来看，新增能力和主要优化都会优先在 2.x 上推进，因此如果你的项目还停留在 1.x，我们建议尽早规划升级。
