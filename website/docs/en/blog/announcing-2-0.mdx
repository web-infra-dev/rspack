# Announcing Rspack 2.0

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

It has been 1 year and 8 months since the official release of Rspack 1.0. During that time, we kept a steady rhythm of a patch release every week and a minor release every two months, gradually aligning with many webpack features such as persistent cache, Layers, and Lazy Compilation, while also filling in a large number of hooks and Node APIs required by plugins. This made it possible for more projects to migrate to Rspack at lower cost.

At the same time, we never stopped pursuing better performance. While continuously optimizing our own algorithms and data structures, we also explored and implemented optimizations such as incremental build, Lazy Barrel, and the native file system watcher, making Rspack a bundler that stays compatible with the webpack ecosystem while delivering stronger build performance.

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

Along the way, Rspack has been recognized by more and more frontend developers. Compared with the 1.0 release, Rspack's weekly downloads have grown by about **48x**. Frameworks, tools, and platforms such as Docusaurus, Nuxt, Nx, Next.js, Storybook, Meteor, Remotion, Addfox, and Vize have also adopted Rspack. This shows that Rspack has gradually grown into a bundler that has been widely validated across the open-source community.

Thanks to all community contributors. Your continuous feedback, discussions, and code contributions keep pushing Rspack forward.

## Try 2.0

### New projects

If you are a new user, we recommend starting directly with a project based on Rspack 2.0. Note that Rspack 2.0 requires **Node.js 20.19+** or **22.12+**.

If you want an out-of-the-box experience powered by Rspack by default, we recommend creating a project with [Rsbuild](https://rsbuild.rs/):

```bash
npm create rsbuild@latest
```

If you prefer to use [Rspack CLI](/api/cli) directly, you can also run:

```bash
npm create rspack@latest
```

Then just follow the prompts to finish setup. For more installation options, see the [Quick start](/guide/start/quick-start) guide.

### Upgrading existing projects

Along with many new features, Rspack 2.0 also removes a number of capabilities that had been deprecated for some time. We recommend starting with the [migration guide](/guide/migration/rspack_1.x), which already summarizes the main breaking changes from v1 to v2 and the corresponding migration paths.

We especially recommend using a Coding Agent together with our [Skills](https://github.com/rstackjs/agent-skills) to handle the migration directly. After installing `rspack-v2-upgrade` with the following command, the agent can combine your project configuration with the official migration rules and complete the upgrade flow. For projects with more configuration and more historical baggage, this can be more efficient than checking items manually one by one.

```
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## Looking ahead

Rspack 2.0 is not only a milestone, but also a new starting point.

In the long run, the Rspack team will gradually shift its focus from "compatibility with webpack" toward **build performance**, **bundle size**, **community conventions**, and **agent collaboration**, while combining the full Rstack toolchain to provide developers with an AI-friendly integrated JavaScript toolchain.

In the short term, the team will focus on the following work:

- Continue improving tree shaking and side effects analysis, and explore function-level module splitting strategies to further reduce bundle size
- Continue optimizing algorithms and data structures across the build pipeline, and connect them with incremental build and persistent cache to improve build speed in different scenarios
- Continue improving the quality and stability of ESM output, and work with other teams in the community to explore better bundler support for the ESM specification
- Continue building [Rstack Skills](https://github.com/rstackjs/agent-skills) to give developers more AI-friendly tools for improving development efficiency and reducing the cost of debugging and output analysis

<span id="faq"></span>
## FAQ

### Does Rspack stop being compatible with webpack after 2.0?

No. Rspack will continue to maintain a limited level of webpack compatibility and will still try to keep migration and adoption smooth for existing projects. The difference is that starting from 2.0, Rspack will no longer treat compatibility as the only goal. When we find approaches that are better for build performance, bundle size, community conventions, or agent support, and those approaches require breaking part of historical compatibility, we will choose to make those changes.

Even so, we will still try to provide detailed transition paths, upgrade notes, and migration guidance to help users understand those changes and move forward with predictable cost.

### After 2.0 is released, will the 1.x line stop being maintained entirely?

No. After 2.0, Rspack's main direction will gradually move to 2.x, but that does not mean the 1.x line will immediately stop receiving maintenance. For projects still using 1.x, we will continue to provide necessary maintenance for a fairly long period in practice, such as critical fixes or migration-period support.

In the long run, however, new capabilities and major optimizations will primarily move forward on 2.x, so if your project is still on 1.x, we recommend planning the upgrade sooner rather than later.

## Faster build performance

In Rspack 2.0, we removed some deprecated features and code, upgraded several outdated dependencies, and used that opportunity to further optimize internal algorithms and data structures. That also brought about roughly a **10%** performance improvement.

<span id="rsc-support"></span>
## Built-in React Server Components support @SyMind

In Rspack 2.0, [React Server Components](/guide/tech/rsc) gain RSC-oriented plugin and compilation support directly in core, rather than relying entirely on upper-layer frameworks to assemble the workflow externally.

Through the `ServerPlugin` and `ClientPlugin` generated by [`rsc.createPlugins()`](/guide/tech/rsc#configure-plugins), Rspack can coordinate both the Server Compiler and Client Compiler pipelines, automatically collect directives such as `"use client"`, `"use server"`, and `"use server-entry"`, generate the manifests required by the RSC runtime, and expose update signals through [`onServerComponentChanges`](/guide/tech/rsc#development-server) when server components change.

At the same time, `builtin:swc-loader` has been enhanced to recognize and transform these RSC directives, allowing React 19-era RSC, Server Actions, client references, and server references to work together within a single build system.

For example:

```js title="rspack.config.mjs"
import { experiments } from '@rspack/core';

const { createPlugins } = experiments.rsc;
const { ServerPlugin, ClientPlugin } = createPlugins();

export default [
  {
    target: 'web',
    plugins: [new ClientPlugin()],
  },
  {
    target: 'node',
    plugins: [new ServerPlugin()],
  },
];
```

<span id="portable-cache"></span>
## Portable cache @jerrykingxyz

In Rspack 1.2, we introduced [persistent cache](/config/cache#persistent-cache), but that cache could not be reused across different machines, different paths, or different environments. That made it difficult to reduce build time through cache reuse in scenarios such as team collaboration and remote builds.

For that reason, in Rspack 2.0 we introduced Portable Cache. You can use [`cache.portable`](/config/cache#cacheportable) to generate and consume cache data that can be copied across environments:

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

In addition to portable cache, Rspack 2.0 also provides [`cache.readonly`](/config/cache#cachereadonly). When enabled, the cache is only read from disk and never written back, which is especially useful in CI because it allows prewarmed cache data to be reused without mutating it.

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

<span id="pure-esm"></span>
## ESM-only packages @chenjiahan

To make Rspack's package format better aligned with the modern Node.js / ESM ecosystem, packages such as `@rspack/core`, `@rspack/cli`, `@rspack/dev-server`, and `@rspack/plugin-react-refresh` are now officially published as ESM-only packages, and the CommonJS builds have been removed as well. This makes module loading and toolchain integration more consistent.

As a result, you need to upgrade Node.js to 20.19+ or 22.12+, which natively support `require(esm)`.

<span id="better-esm"></span>
## Better ESM output

For library bundling scenarios, ESM output quality is especially important. The cleaner the ESM output is, the better it is for static analysis, and the more effective downstream code splitting and tree shaking can be.

In Rspack 1.6 we shipped the experimental [`rspack.experiments.EsmLibraryPlugin`](https://v1.rspack.rs/plugins/rspack/esm-library-plugin), which replaced the module-concatenation-based ESM output pipeline with a dedicated one. Over the following minor releases it was hardened and made responsible for more parts of the pipeline, and in 2.0 it is fully stabilized. The experimental plugin entry is gone — you enable the same pipeline by setting [`output.library.type`](/config/output#outputlibrarytype) to `'modern-module'`:

```js title="rspack.config.mjs"
export default {
  output: {
    library: {
      type: 'modern-module',
    },
  },
};
```

Compared with the generic [`output.module`](/config/output#outputmodule) path — which stays on the regular chunk/runtime renderer and then turns entry exports into `export` statements — `modern-module` takes over the full library rendering path. In particular:

- It uses its own scope-hoisting algorithm instead of [`optimization.concatenateModules`](/config/optimization#optimizationconcatenatemodules), so multiple modules can be concatenated into a single ESM output without dragging in Rspack's runtime wrappers.
- It renders chunks directly as ESM files and links them together with real `import` / `export` statements, so consumers can keep statically analyzing the output even after [code splitting](/guide/optimization/code-splitting).
- It deduplicates modules that would otherwise appear in multiple chunks, reducing redundancy when a library is split into several entries.
- It unlocks [`output.library.preserveModules`](/config/output#outputlibrarypreservemodules), which preserves the original source directory structure in the output — useful for publishing libraries whose consumers expect one output file per source file.

At the configuration level, `modern-module` also implies a set of defaults that are otherwise tedious to wire up by hand — [`output.module`](/config/output#outputmodule) is set to `true`, `output.chunkLoading` is set to `'import'`, and [`splitChunks`](/plugins/webpack/split-chunks-plugin) defaults are widened so that split output remains clean. For a broader walk-through of ESM scenarios and configuration, see the [ESM guide](/guide/features/esm).

<span id="import-meta-properties"></span>
## Better import.meta support @ahabhgk

On the source side, Rspack 2.0 continues improving `import.meta` support. For unknown `import.meta` properties, you can configure [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) so that they are not resolved too early at compile time and are instead preserved for runtime evaluation.

For example, if you want unknown `import.meta` properties to be preserved until runtime, you can configure:

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

In addition, Rspack 2.0 also adds support for capabilities such as [`import.meta.main`](/api/runtime-api/module-variables#importmetamain), [`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename), and [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname).

<span id="import-defer"></span>
## Better deferred import support @ahabhgk

Rspack first added support for the `import defer` syntax in 1.6, and in 2.0 that capability continues to evolve toward a more complete form. In addition to the original `import defer * as foo from './foo'`, Rspack later added support for `import.defer()`, allowing the "import now, execute later" model to cover more real-world usage. This gives developers more precise control over module execution timing and side effects.

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

<span id="dev-server"></span>
## A lighter development server @chenjiahan

In 2.0, `@rspack/dev-server` is no longer a default dependency of `@rspack/cli`, since some scenarios do not need a development server at all. If your project uses `rspack dev` or `rspack serve`, you now need to install it manually:

```bash
pnpm add -D @rspack/dev-server
```

At the same time, to further slim down `@rspack/dev-server`, it now introduces `@rspack/dev-middleware` to replace `webpack-dev-middleware`, and uses `connect-next` instead of Express v4.

<span id="subpath-alias-import"></span>
## Support for `#/` subpath alias imports @stormslowly

In module resolution, Rspack 2.0 supports `#/`-style subpath alias imports. This means you can directly reuse the `imports` field in `package.json` to organize intra-package path mapping, instead of maintaining a separate alias configuration.

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

<span id="pnp-manifest"></span>
## Support for resolve.pnpManifest @stormslowly

For Yarn PnP scenarios, the Rspack 2.0 resolver adds support for the `resolve.pnpManifest` option, which lets you explicitly specify the path to the PnP manifest file. This gives you more fine-grained control over how PnP metadata is read and resolved in complex package-management and monorepo environments, instead of relying entirely on auto-detection:

```js title="rspack.config.mjs"
import path from 'node:path';

export default {
  resolve: {
    pnp: true,
    pnpManifest: path.resolve(import.meta.dirname, '.pnp.cjs'),
  },
};
```

## Optional Module Federation runtime @chenjiahan

In 2.0, `@module-federation/runtime-tools` has also been changed into an optional dependency, and its compatible version range has been widened to accept newer runtime versions as well.

If you use Module Federation, you now need to install `@module-federation/runtime-tools` manually:

```bash
pnpm add @module-federation/runtime-tools
```

<span id="mf-share-treeshaking"></span>
## Module Federation shared dependency pruning @2heal1

Rspack 2.0 further improves pruning for shared dependencies in [Module Federation](/guide/features/module-federation), allowing finer-grained export-level optimization on `shared` dependencies and removing parts of shared packages that are not actually used. This further reduces output size in micro-frontend scenarios.

It supports both `runtime-infer` and `server-calc` modes, targeting two different engineering strategies: runtime inference and deployment-side secondary calculation.

For example:

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

## Other feature updates

<span id="detect-syntax"></span>
### Built-in SWC loader syntax detection @chenjiahan

When one rule handles different file types such as `.js`, `.jsx`, `.ts`, and `.tsx` together, [`detectSyntax` in `builtin:swc-loader`](/guide/features/builtin-swc-loader#detectsyntax) can infer parser configuration from the file extension automatically, helping you avoid repeated configuration.

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

<span id="automatic-targets"></span>
### Automatic inheritance of target settings @ahabhgk

The target-environment options of [`builtin:swc-loader`](/guide/features/builtin-swc-loader), [`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader), [`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin), and [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) now inherit from the top-level [`target`](/config/target) configuration by default. Developers no longer need to keep loader, plugin, and top-level target settings in sync manually, which reduces configuration duplication and avoids mismatches between transform targets and minification targets.

<span id="hashed-module-ids-plugin"></span>
### Support for HashedModuleIdsPlugin

`HashedModuleIdsPlugin` is also supported in 2.0. If you want a short, stable, and unique id derived from the relative path of a module, you can use this plugin:

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

<span id="enforce-size-threshold"></span>
### Support for splitChunks.enforceSizeThreshold

The [`splitChunks.enforceSizeThreshold`](/plugins/webpack/split-chunks-plugin) option is also supported in 2.0. It sets a size threshold at which chunk splitting is enforced regardless of [`splitChunks.maxAsyncRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxasyncrequests), [`splitChunks.maxInitialRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxinitialrequests), and `splitChunks.minRemainingSize`. Without this option, those request limits can prevent a large cache group from being split out even when splitting would clearly be desirable; once the group size crosses the threshold, those limits are skipped so the split still happens.

Following webpack's defaults, Rspack defaults `enforceSizeThreshold` to `50000` in production mode and `30000` in other modes, and also accepts a per-type object to target specific module types such as JavaScript and CSS:

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      // single number applies to all source types
      enforceSizeThreshold: 50000,
      // or per source type
      // enforceSizeThreshold: { javascript: 50000, css: 30000 },
    },
  },
};
```

The option can also be set per cache group, so you can keep a strict global request budget while letting a specific group (for example `defaultVendors`) split more aggressively once it grows past a chosen size. Other constraints such as [`minSize`](/plugins/webpack/split-chunks-plugin#splitchunksminsize), [`maxSize`](/plugins/webpack/split-chunks-plugin#splitchunksmaxsize), and [`minChunks`](/plugins/webpack/split-chunks-plugin#splitchunksminchunks) continue to apply.

<span id="better-tree-shaking"></span>
### Better tree shaking @ahabhgk

2.0 introduces `experiments.pureFunctions`, which allows finer-grained analysis of cross-module pure function calls through `/*#__NO_SIDE_EFFECTS__*/` annotations or the [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) option.

At the same time, some CommonJS imports and exports that were not recognized correctly in the past can now be parsed properly in 2.0, further improving [tree shaking](/guide/optimization/tree-shaking).

## Other changes

- [`module.parser.javascript.exportsPresence`](/config/module#moduleparserjavascriptexportspresence) now defaults to `error`, so missing exports now fail with errors instead of warnings.
- `module.parser.javascript.strictExportPresence` has been removed, and missing-export behavior is now controlled by [`module.parser.javascript.exportsPresence`](/config/module#moduleparserjavascriptexportspresence).
- [`module.parser.javascript.requireAlias`](/config/module#moduleparserjavascriptrequirealias) now defaults to `false`, so renamed `require` calls are no longer parsed.
- [`builtin:swc-loader`](/guide/features/builtin-swc-loader) no longer reads `.swcrc`, and SWC configuration should be moved into loader options.
- The `rspackExperiments.collectTypeScriptInfo` option of [`builtin:swc-loader`](/guide/features/builtin-swc-loader) has been removed, and TypeScript information collection is now configured through [`collectTypeScriptInfo`](/guide/features/builtin-swc-loader#collecttypescriptinfo).
- `builtin:swc-loader.rspackExperiments.import` has been moved to the top-level [`transformImport`](/guide/features/builtin-swc-loader#transformimport).
- `output.libraryTarget`, `output.libraryExport`, `output.umdNamedDefine`, and `output.auxiliaryComment` have all been moved under [`output.library.*`](/config/output#outputlibrary).
- [`output.chunkLoadingGlobal`](/config/output#outputchunkloadingglobal) now defaults to `rspackChunk${output.uniqueName}`.
- [`output.hotUpdateGlobal`](/config/output#outputhotupdateglobal) now defaults to `rspackHotUpdate${output.uniqueName}`.
- [`output.bundlerInfo.force`](/config/output#outputbundlerinfo) now defaults to `false`, and bundler information is no longer injected unconditionally.
- [`output.trustedTypes.policyName`](/config/output#outputtrustedtypes) now falls back to `'rspack'` instead of `'webpack'` when `output.uniqueName` is not set.
- `output.charset` has been removed, and modern browsers treat scripts as UTF-8.
- `optimization.removeAvailableModules` has been removed.
- [`stats.toJson()`](/api/javascript-api/stats#tojson) no longer returns `modules`, `assets`, `chunks`, `chunkGroups`, and `entryPoints` by default.
- `profile` and `stats.profile` have been removed, and [Rsdoctor](/guide/optimization/use-rsdoctor) is now recommended for performance analysis.
- [`WarnCaseSensitiveModulesPlugin`](/plugins/webpack/case-sensitive-plugin) has been replaced by [`CaseSensitivePlugin`](/plugins/webpack/case-sensitive-plugin).
- The `getHooks` alias on [`HtmlRspackPlugin`](/plugins/rspack/html-rspack-plugin) and [`RuntimePlugin`](/api/plugin-api/runtime-plugin-hooks) has been removed, and you should use `getCompilationHooks` instead.
- The `sri` configuration on [`HtmlRspackPlugin`](/plugins/rspack/html-rspack-plugin) and [`RuntimePlugin`](/api/plugin-api/runtime-plugin-hooks) has been removed. SRI-related capability should now use [`SubresourceIntegrityPlugin`](/plugins/rspack/subresource-integrity-plugin).
- The `draft` option of [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin.mdx) has been removed in favor of `drafts`, and `cssHeadDataCompression` has also been removed.
- JavaScript requests no longer include `.wasm` in the default [`resolve.extensions`](/config/resolve#resolveextensions).
- CSS `@import` no longer includes `webpack` in the default [`conditionNames`](/config/resolve#resolveconditionnames).
- [`devtool`](/config/devtool) now defaults to `cheap-module-source-map` in development and to `false` in CLI production mode.
- [`resolve.roots`](/config/resolve#resolveroots) now defaults to `[]`.
- Deprecated runtime module properties such as `constructorName` and `moduleIdentifier` have been removed.
- [`ProgressPlugin`](/plugins/webpack/progress-plugin) custom handlers now receive a structured `info` object instead of `items: string[]`.
- `NormalModule.getCompilationHooks(compilation).readResourceForScheme` has been removed, and `readResource.for(scheme)` should be used instead.
- `module.unsafeCache` has been removed.
- `experiments.css` has been removed, and [CSS](/guide/tech/css) capability is now available by default.
- [`experiments.asyncWebAssembly`](/config/experiments#experimentsasyncwebassembly) now defaults to `true`.
- `experiments.cache` has been moved to the top-level [`cache`](/config/cache).
- `experiments.rspackFuture` has been removed, and `bundlerInfo` has moved to [`output.bundlerInfo`](/config/output#outputbundlerinfo).
- `experiments.incremental` has been moved to the top-level [`incremental`](/config/incremental).
- `experiments.layers` has been removed, and [layers](/guide/features/layer) are now available by default.
- `experiments.topLevelAwait` has been removed, and top-level await parsing is now available by default.
- `experiments.lazyCompilation` has been moved to the top-level [`lazyCompilation`](/config/lazy-compilation).
- `experiments.lazyCompilationMiddleware` has been moved to the top-level [`lazyCompilationMiddleware`](/guide/features/lazy-compilation#integrating-with-custom-server).
- `experiments.lazyBarrel` has been removed, and [lazy barrel](/guide/optimization/lazy-barrel) optimization is now enabled by default.
- `experiments.typeReexportsPresence` has been removed, and its behavior is now controlled through [`module.parser.javascript.typeReexportsPresence`](/config/module#moduleparserjavascripttypereexportspresence).
- `experiments.inlineConst` has been removed, and constant inlining is now controlled through [`optimization.inlineExports`](/config/optimization#optimizationinlineexports).
- `experiments.inlineEnum` has been removed, and enum inlining is now controlled through [`optimization.inlineExports`](/config/optimization#optimizationinlineexports) together with [`collectTypeScriptInfo.exportedEnum`](/guide/features/builtin-swc-loader#collecttypescriptinfoexportedenum).
- `experiments.outputModule` has been removed, and [ESM output](/guide/features/esm) is now controlled through [`output.module`](/config/output#outputmodule).
- `experiments.parallelLoader` has been removed, and [parallel loader](/config/module-rules#rulesuseparallel) capability is now available by default.
