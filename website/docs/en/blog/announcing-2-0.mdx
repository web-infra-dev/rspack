# Announcing Rspack 2.0

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

It has been 1 year and 8 months since the official release of Rspack 1.0.

During that time, we kept a steady cadence of one patch release every week and one minor release every two months. Rspack gradually filled in webpack features such as persistent cache, Layers, and Lazy Compilation, and also completed many of the hooks and Node APIs that plugins rely on. This made it easier for more projects to migrate to Rspack with lower and more predictable cost.

At the same time, we continued investing in performance. Alongside ongoing improvements to internal algorithms and data structures, we also introduced capabilities such as incremental build, Lazy Barrel, and the native file system watcher. Rspack has therefore gradually grown into a bundler that stays compatible with the webpack ecosystem while delivering stronger build performance.

![Rspack 2.0 npm downloads](https://assets.rspack.rs/rspack/assets/rspack-v2-0-npm-downloads.png)

Along the way, Rspack has also been recognized by more and more frontend developers. Compared with the 1.0 release, Rspack's weekly downloads have grown by about **48x**. Frameworks, tools, and platforms such as Docusaurus, Nuxt, Nx, Next.js, Storybook, Meteor, Remotion, Addfox, and Vize have also adopted Rspack.

That means Rspack has now been validated by more real-world projects and is gradually becoming a bundler the community can rely on in production.

![Rspack 2.0 repo stats](https://assets.rspack.rs/rspack/assets/rspack-v2-0-repo-stats.png)

Community support has continued to grow as well. Since the release of 1.0, the Rspack repository has received **4,728 pull requests** and **1,052 issues**, with **164 contributors** participating in the project.

Thanks to all community contributors. Ongoing feedback, discussions, and code contributions are what keep moving Rspack forward.

## Try 2.0

### New projects

If you are new to Rspack, we recommend starting with a new project based on Rspack 2.0. Note that Rspack 2.0 requires **Node.js 20.19+** or **22.12+**.

If you want an out-of-the-box experience powered by Rspack by default, we recommend creating a project with [Rsbuild](https://rsbuild.rs/):

```bash
npm create rsbuild@latest
```

If you prefer to use [Rspack CLI](/api/cli) directly, you can also run:

```bash
npm create rspack@latest
```

Then just follow the prompts to finish setup. For more installation options, see the [Quick start](/guide/start/quick-start) guide.

### Upgrading existing projects

Rspack 2.0 introduces many new capabilities, but it also removes a number of features that had already been deprecated. For existing projects, we recommend starting with the [migration guide](/guide/migration/rspack_1.x). It already summarizes the main breaking changes from v1 to v2 and the corresponding migration paths.

If you are using a Coding Agent with Skills support, we also recommend using it for the migration. After installing `rspack-v2-upgrade`, the agent can help complete the upgrade flow by combining your project configuration with the official migration rules. For projects with more configuration and more historical baggage, this is often more efficient than checking items manually one by one.

```
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## Looking ahead

Rspack 2.0 is a milestone, and also a new starting point.

In the long run, the Rspack team will gradually shift its focus from "compatibility with webpack" toward **build performance**, **bundle size**, **community conventions**, and **agent collaboration**, while also combining the full Rstack toolchain to provide developers with an AI-friendly integrated JavaScript toolchain.

In the near term, the team will focus on the following areas:

- Continue improving tree shaking and side effects analysis, and explore function-level module splitting strategies to further reduce bundle size
- Continue optimizing algorithms and data structures across the build pipeline, and connect them with incremental build and persistent cache to improve build speed in different scenarios
- Continue improving the quality and stability of ESM output, and work with other teams in the community to explore better bundler support for the ESM specification
- Continue building [Rstack Skills](https://github.com/rstackjs/agent-skills) to give developers more AI-friendly tools for improving development efficiency and reducing the cost of debugging and output analysis

## FAQ \{#faq}

### Does Rspack stop being compatible with webpack after 2.0?

No. Rspack will continue to maintain a limited level of webpack compatibility and will still try to keep migration and adoption smooth for existing projects.

The change is that, starting with 2.0, compatibility is no longer the only goal. When we find approaches that are better for build performance, bundle size, community conventions, or agent support, and those approaches require breaking part of historical compatibility, we will choose to make those changes.

Even so, we will continue to provide clear transition paths, upgrade notes, and migration guidance so users can understand those changes and move forward with predictable cost.

### After 2.0 is released, will the 1.x line stop being maintained entirely?

No. After 2.0, Rspack's main direction will gradually move to 2.x, but that does not mean the 1.x line will stop receiving maintenance right away. For projects still using 1.x, we will continue to provide necessary maintenance for a fairly long period in practice, such as critical fixes or migration-period support.

In the long run, however, new capabilities and major optimizations will primarily move forward on 2.x, so if your project is still on 1.x, we recommend planning the upgrade sooner rather than later.

## Faster build performance \{#build-performance}

![Rspack 2.0 performance comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-perf-comparison.png)

In Rspack 2.0, we removed some deprecated features and legacy code, and upgraded several outdated dependencies. That created room for further optimization of internal algorithms and data structures, and brought about roughly a **10%** performance improvement.

## Built-in React Server Components support \{#rsc-support}

Rspack v2.0.0 and later include built-in support for React Server Components.

This support includes:

1. **Framework-agnostic support**: Universal underlying RSC build capabilities without depending on a specific meta-framework.
2. **Compile-time checks**: API calls that violate RSC rules can be intercepted during compilation, before they become runtime errors.
3. **CSS support**: Styles from both server and client components can be collected at compile time and injected during rendering.
4. **HMR support**: Hot updates work for both server and client components.

There are currently two ways to use it:

- **Use Rsbuild**: Rsbuild provides out-of-the-box support for React Server Components and lets you create an RSC project quickly. See [rsbuild-plugin-rsc](https://github.com/rstackjs/rsbuild-plugin-rsc) for details.
- **Configure Rspack manually**: You can also follow the current documentation and add the React Server Components configuration yourself.

The [rspack-rsc-examples](https://github.com/rstackjs/rspack-rsc-examples) repository contains complete examples of building RSC applications with Rspack.

## Portable cache @jerrykingxyz \{#portable-cache}

In Rspack 1.2, we introduced [persistent cache](/config/cache#persistent-cache). However, that cache could not be reused across different machines, different paths, or different environments, which made cache reuse much less useful for scenarios such as team collaboration and remote builds.

In Rspack 2.0, we took the next step and introduced Portable Cache. You can use [`cache.portable`](/config/cache#cacheportable) to generate and consume cache data that can be copied across environments:

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

Rspack 2.0 also provides [`cache.readonly`](/config/cache#cachereadonly). When enabled, the cache is only read from disk and never written back. This is especially useful in CI, where prewarmed cache data can be reused without being modified.

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

## ESM-only packages @chenjiahan \{#pure-esm}

To make Rspack's package format better aligned with the modern Node.js / ESM ecosystem, packages such as `@rspack/core`, `@rspack/cli`, `@rspack/dev-server`, and `@rspack/plugin-react-refresh` are now officially published as ESM-only packages, and the CommonJS builds have been removed as well. This makes module loading and toolchain integration more consistent.

As a result, you need to upgrade Node.js to 20.19+ or 22.12+, which natively support `require(esm)`.

## Better ESM output \{#better-esm}

For library builds, ESM output quality matters a lot. The cleaner the output is, the easier it is to analyze statically, and the better downstream tools can perform code splitting and tree shaking.

In Rspack 1.6, we introduced the experimental [`rspack.experiments.EsmLibraryPlugin`](https://v1.rspack.rs/plugins/rspack/esm-library-plugin), replacing the earlier module-concatenation-based ESM output path with a dedicated one. Over the following minor releases, that path was refined and gradually took over more stages of the output pipeline. In 2.0, it is now fully stabilized.

At the same time, the experimental plugin entry has been removed. To enable the same path now, you only need to set [`output.library.type`](/config/output#outputlibrarytype) to `'modern-module'`:

```js title="rspack.config.mjs"
export default {
  output: {
    library: {
      type: 'modern-module',
    },
  },
};
```

Compared with the generic [`output.module`](/config/output#outputmodule) path, `modern-module` takes over the full library rendering flow. The former still uses the standard chunk/runtime renderer and then turns entry exports into `export` statements. `modern-module`, by contrast, follows a dedicated path designed for library output. In particular:

- It uses its own scope-hoisting algorithm instead of [`optimization.concatenateModules`](/config/optimization#optimizationconcatenatemodules), so multiple modules can be concatenated into a single ESM output without dragging in Rspack's runtime wrappers.
- It renders chunks directly as ESM files and links them together with real `import` / `export` statements, so consumers can keep statically analyzing the output even after [code splitting](/guide/optimization/code-splitting).
- It deduplicates modules that would otherwise appear in multiple chunks, reducing redundancy when a library is split into several entries.
- It unlocks [`output.library.preserveModules`](/config/output#outputlibrarypreservemodules), which preserves the original source directory structure in the output — useful for publishing libraries whose consumers expect one output file per source file.

At the configuration level, `modern-module` also implies a set of defaults that would otherwise be tedious to wire up by hand: [`output.module`](/config/output#outputmodule) is set to `true`, `output.chunkLoading` is set to `'import'`, and [`splitChunks`](/plugins/webpack/split-chunks-plugin) defaults are widened so that split output remains clean. For a broader walk-through of ESM scenarios and configuration, see the [ESM guide](/guide/features/esm).

## Better import.meta support @ahabhgk \{#import-meta-properties}

On the source side, Rspack 2.0 continues improving `import.meta` support. For unknown `import.meta` properties, you can configure [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta) so they are not resolved too early at compile time and are instead preserved for runtime evaluation.

For example, if you want unknown `import.meta` properties to be preserved until runtime, you can configure:

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

In addition, Rspack 2.0 also adds support for capabilities such as [`import.meta.main`](/api/runtime-api/module-variables#importmetamain), [`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename), and [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname).

## Better deferred import support @ahabhgk \{#import-defer}

Rspack first added support for the `import defer` syntax in 1.6. In 2.0, that support moves one step closer to a more complete form. In addition to the original `import defer * as foo from './foo'`, Rspack now also supports `import.defer()`, allowing the "import now, execute later" model to cover more real-world usage.

This gives developers more precise control over when modules execute and when side effects are triggered.

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

## A lighter development server @chenjiahan \{#dev-server}

In 2.0, `@rspack/dev-server` is no longer a default dependency of `@rspack/cli`, because not every project needs a development server. If your project uses `rspack dev` or `rspack serve`, you now need to install it manually:

```bash
pnpm add -D @rspack/dev-server
```

At the same time, to further slim down `@rspack/dev-server`, we introduced `@rspack/dev-middleware` to replace `webpack-dev-middleware`, and `connect-next` to replace Express v4.

## Support for `#/` subpath alias imports @stormslowly \{#subpath-alias-import}

In module resolution, Rspack 2.0 supports `#/`-style subpath alias imports. This means you can directly reuse the `imports` field in `package.json` to organize intra-package path mapping, instead of maintaining a separate alias configuration.

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

In 2.0, `@module-federation/runtime-tools` has also been changed into an optional dependency, and its compatible version range has been widened to accept newer runtime versions as well.

If you use Module Federation, you now need to install `@module-federation/runtime-tools` manually:

```bash
pnpm add @module-federation/runtime-tools
```

## Module Federation shared treeshaking \{#mf-share-treeshaking}

Rspack 2.0 further improves shared treeshaking for [Module Federation](/guide/features/module-federation). It can optimize `shared` dependencies at the export level and remove parts that are not actually used, which helps reduce the size of shared bundles in micro-frontend scenarios.

In the Module Federation shared used in Rspack 1.x, once a dependency was declared in `shared`, the runtime usually had to prepare the full package. Even if an application only used a small subset of its exports, the entire dependency was often pulled in. For libraries with many exports, such as `antd` or `lodash-es`, this could noticeably increase the size of shared bundles.

Rspack 2.0 integrates this capability directly into `ModuleFederationPlugin`. When `treeshaking` is enabled for a `shared` entry, Rspack generates a treeshaking result for that shared dependency and lets the runtime load it first. If the current request cannot be satisfied by that result, the runtime falls back to the full dependency to preserve stable behavior.

It supports both `runtime-infer` and `server-calc` modes:

- `runtime-infer` is suitable for quick adoption and local validation. The runtime reuses a treeshaking shared bundle when possible, and falls back to the full dependency when the required exports are not available. You can also declare common exports ahead of time through `usedExports` to improve hit rates.
- `server-calc` is better suited to production deployments with multiple consuming applications. You can aggregate the exports actually used by consumers during deployment and generate a more stable shared dependency treeshaking result.

For component libraries and other large shared packages, this turns "share the full package" into "share what is actually used", reducing unnecessary code loading. For more details on configuration and behavior, see the [Shared Treeshaking guide](https://module-federation.io/guide/performance/shared-tree-shaking).

For example:

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

## Other feature updates

### Built-in SWC loader syntax detection @chenjiahan \{#detect-syntax}

When one rule handles different file types such as `.js`, `.jsx`, `.ts`, and `.tsx` together, [`detectSyntax` in `builtin:swc-loader`](/guide/features/builtin-swc-loader#detectsyntax) can infer parser configuration from the file extension automatically, helping you avoid repeated configuration.

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

### CSS parser `resolveImport` option \{#resolve-import}

Rspack 2.0 exposes a [`resolveImport`](/config/module#moduleparsercssautoresolveimport) option on the CSS parser, available on `css`, `css/auto`, and `css/module`. It controls whether `@import` at-rules are resolved and inlined at build time.

When set to `true` (the default), `@import` rules are processed as dependencies and replaced with the imported content. Setting it to `false` preserves the `@import` statement verbatim in the output and leaves resolution to the browser or a downstream tool.

The option can also take a callback. Rspack calls it for every `@import` rule and passes the URL, the path of the importing file, and the full set of at-rule conditions (`media`, `supports`, `layer`). This makes it easy to handle imports selectively. For example, you can inline local stylesheets while keeping remote URLs as real `@import` statements for the browser to fetch:

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      'css/auto': {
        resolveImport: ({ url }) => !/^https?:\/\//.test(url),
      },
    },
  },
};
```

### Automatic inheritance of target settings @ahabhgk \{#automatic-targets}

The target-environment options of [`builtin:swc-loader`](/guide/features/builtin-swc-loader), [`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader), [`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin), and [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) now inherit from the top-level [`target`](/config/target) configuration by default.

This means developers no longer need to keep loader, plugin, and top-level target settings in sync by hand. It reduces configuration duplication and also avoids mismatches between transform targets and minification targets.

### Support for HashedModuleIdsPlugin \{#hashed-module-ids-plugin}

`HashedModuleIdsPlugin` is also supported in 2.0. If you want a short, stable, and unique id derived from the relative path of a module, you can use this plugin:

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

### Support for splitChunks.enforceSizeThreshold \{#enforce-size-threshold}

The [`splitChunks.enforceSizeThreshold`](/plugins/webpack/split-chunks-plugin) option is also supported in 2.0. It defines a size threshold at which chunk splitting is enforced. Once a module group exceeds that threshold, limits such as [`splitChunks.maxAsyncRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxasyncrequests), [`splitChunks.maxInitialRequests`](/plugins/webpack/split-chunks-plugin#splitchunksmaxinitialrequests), and `splitChunks.minRemainingSize` are ignored so the split can still happen.

Without this option, request limits can prevent a large cache group from being split out even when splitting would clearly be the better outcome. With the threshold in place, Rspack skips those limits once the group becomes large enough and continues with the split.

Following webpack's defaults, Rspack sets `enforceSizeThreshold` to `50000` in production mode and `30000` in other modes. It also accepts a per-type object so you can target specific module types such as JavaScript and CSS:

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      // single number applies to all source types
      enforceSizeThreshold: 50000,
      // or per source type
      // enforceSizeThreshold: { javascript: 50000, css: 30000 },
    },
  },
};
```

The option can also be set per cache group. This lets you keep a strict global request budget while allowing a specific group (for example `defaultVendors`) to split more aggressively once it grows past a chosen size.

Other constraints such as [`minSize`](/plugins/webpack/split-chunks-plugin#splitchunksminsize), [`maxSize`](/plugins/webpack/split-chunks-plugin#splitchunksmaxsize), and [`minChunks`](/plugins/webpack/split-chunks-plugin#splitchunksminchunks) continue to apply.

### Better tree shaking @ahabhgk \{#better-tree-shaking}

2.0 introduces `experiments.pureFunctions`, which allows finer-grained analysis of cross-module pure function calls through `/*#__NO_SIDE_EFFECTS__*/` annotations or the [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions) option.

At the same time, some CommonJS imports and exports that were not recognized correctly in the past can now be parsed properly in 2.0, which further improves [tree shaking](/guide/optimization/tree-shaking).

- [`resolve.roots`](/config/resolve#resolveroots) now defaults to `[]`.
- Deprecated runtime module properties such as `constructorName` and `moduleIdentifier` have been removed.
- [`ProgressPlugin`](/plugins/webpack/progress-plugin) custom handlers now receive a structured `info` object instead of `items: string[]`.
- `NormalModule.getCompilationHooks(compilation).readResourceForScheme` has been removed, and `readResource.for(scheme)` should be used instead.
- `module.unsafeCache` has been removed.
- `experiments.css` has been removed, and [CSS](/guide/tech/css) capability is now available by default.
- [`experiments.asyncWebAssembly`](/config/experiments#experimentsasyncwebassembly) now defaults to `true`.
- `experiments.cache` has been moved to the top-level [`cache`](/config/cache).
- `experiments.rspackFuture` has been removed, and `bundlerInfo` has moved to [`output.bundlerInfo`](/config/output#outputbundlerinfo).
- `experiments.incremental` has been moved to the top-level [`incremental`](/config/incremental).
- `experiments.layers` has been removed, and [layers](/guide/features/layer) are now available by default.
- `experiments.topLevelAwait` has been removed, and top-level await parsing is now available by default.
- `experiments.lazyCompilation` has been moved to the top-level [`lazyCompilation`](/config/lazy-compilation).
- `experiments.lazyCompilationMiddleware` has been moved to the top-level [`lazyCompilationMiddleware`](/guide/features/lazy-compilation#integrating-with-custom-server).

- `experiments.inlineConst` has been removed, and constant inlining is now controlled through [`optimization.inlineExports`](/config/optimization#optimizationinlineexports).
- `experiments.inlineEnum` has been removed, and enum inlining is now controlled through [`optimization.inlineExports`](/config/optimization#optimizationinlineexports) together with [`collectTypeScriptInfo.exportedEnum`](/guide/features/builtin-swc-loader#collecttypescriptinfoexportedenum).
- `experiments.outputModule` has been removed, and [ESM output](/guide/features/esm) is now controlled through [`output.module`](/config/output#outputmodule).
- `experiments.parallelLoader` has been removed, and [parallel loader](/config/module-rules#rulesuseparallel) capability is now available by default.
