# Announcing Rspack 2.0

![Rspack 2.0 Banner](https://assets.rspack.rs/rspack/assets/rspack-v2-0-banner.png)

---

We are pleased to announce that Rspack 2.0 is now available.

This is the first major release after Rspack 1.0, arriving 1 year and 8 months after the 1.0 release. Throughout the 1.x series, our primary goal was to close the gap with webpack in features and ecosystem compatibility, so that more projects could migrate to Rspack with lower cost. To support that goal, we gradually improved module capabilities, dev server behavior, caching, ESM support, and other engineering features, making Rspack a fast bundling option for existing projects while remaining compatible with the webpack ecosystem.

With the release of 2.0, Rspack's direction also expands. Going forward, while keeping compatibility with most of webpack, Rspack will explore better solutions in build performance, bundle size, community conventions, and AI collaboration. In other words, Rspack is no longer only focused on being compatible; it will also keep pushing for implementations and workflows that are better suited to modern JavaScript engineering.

Highlights include:

- Continued growth in ecosystem adoption
- Faster builds and further improvements in bundle size
- [Pure ESM](/guide/migration/rspack_1.x#pure-esm-packages), better ESM output, and more complete [`import.meta`](/api/runtime-api/module-variables) behavior
- A lighter, more modular [`@rspack/dev-server` 2.0](/guide/migration/rspack_1.x#development-server)
- [Portable Cache](/config/cache#cacheportable), [automatic targets inheritance](/guide/migration/rspack_1.x#derive-loader-and-plugin-targets-from-target), and clearer peer dependency boundaries
- Stronger [tree shaking](/guide/optimization/tree-shaking), [`import defer`](/blog/announcing-1-6#support-for-import-defer), MF share treeshaking, and built-in [RSC support](/guide/tech/rsc)

## Try 2.0

### New projects

If you are new to Rspack, the easiest way to try 2.0 is to start with a fresh project. Note that Rspack 2.0 requires **Node.js 20.19+** or **22.12+**.

If you want an out-of-the-box experience powered by Rspack, we recommend creating a project with Rsbuild:

```bash
npm create rsbuild@latest
```

If you prefer to use Rspack CLI directly, you can also run:

```bash
npm create rspack@latest
```

Then just follow the interactive prompts. For more installation options, see the [Quick start](/guide/start/quick-start) guide.

### Upgrading existing projects

Rspack 2.0 is a major release. In addition to new features, it also removes a number of previously deprecated capabilities. We recommend starting with the [migration guide](/guide/migration/rspack_1.x), which summarizes the main incompatible changes from v1 to v2 and the corresponding migration paths.

If you are using a coding agent that supports Skills, AI-assisted migration can also help. As described in the migration guide, after installing the `rspack-v2-upgrade` skill, the agent can combine your project configuration with the official migration rules to guide the upgrade process. For projects with heavier historical configuration, this can be more efficient than checking items manually one by one.

```bash
npx skills add rstackjs/agent-skills --skill rspack-v2-upgrade
```

## Key metrics

### Better performance

![Rspack 2.0 performance comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-perf-comparison.png)

Across the 1.x series, we invested heavily in performance-related capabilities, and many of them were first introduced experimentally before being gradually refined and stabilized. For example, persistent cache, incremental build, the native file watcher, lazy compilation, and barrel optimization all helped reduce overhead from cache misses, redundant work, file-change response, and unnecessary module resolution and building.

Together, these improvements gave the 1.x series a major performance lift compared with the original 1.0 release. In 2.0, we then went further by optimizing internal data structures and algorithms, bringing another roughly 10% improvement on top of that. Using the `react-1k` benchmark from build-tools-performance as an example, Rspack 2.0 shows clear improvements in both Build and HMR compared with Rspack 1.0. In large projects, this translates into shorter waiting time and faster feedback.

### Better bundle size

![Rspack 2.0 bundle size comparison](https://assets.rspack.rs/rspack/assets/rspack-v2-0-bundle-size-comparison.png)

For bundle size, the main focus throughout 1.x was improving two things: how accurately Rspack can identify removable code, and how well the generated output supports static analysis. To move in that direction, we continued improving [tree shaking](/guide/optimization/tree-shaking), introduced and later enabled [barrel file optimization](/blog/announcing-1-5#barrel-file-optimization) by default, and added capabilities such as [const inline optimization](/blog/announcing-1-5#const-inline-optimization), TypeScript enum inlining, and [ESM output](/guide/features/esm).

Together, these changes extend Rspack's bundle pruning from module-level optimization toward export-level, expression-level, and in some cases function-level optimization. In Rspack 2.0, capabilities such as `experiments.pureFunctions` continue that direction. Bundle optimization will remain an important focus, especially around higher-quality tree shaking, more precise code generation, and more modern module output.

## Feature updates

### Pure ESM

At the package distribution level, Rspack 2.0 also moves closer to the ESM standard. Rspack 2.0 now publishes `@rspack/core`, `@rspack/cli`, `@rspack/dev-server`, and `@rspack/plugin-react-refresh` as [pure ESM packages](/guide/migration/rspack_1.x#pure-esm-packages), with CommonJS builds removed. This keeps Rspack aligned with the modern Node.js / ESM ecosystem and simplifies follow-up work around module loading and toolchain integration.

One direct benefit of moving to pure ESM is a more unified package distribution model: there is no longer a need to maintain both ESM and CommonJS outputs and the branching logic that comes with them. That reduces maintenance cost, avoids behavior differences between module formats, and makes things like `import.meta`, modern module loading, conditional exports, and integration with upper-layer tooling feel more natural.

At the same time, Rspack 2.0 now requires a minimum Node.js version of **20.19+** or **22.12+**, and Node.js 18 is no longer supported. One important reason for this change is that Rspack can rely on Node.js native `require(esm)` support to load ESM modules directly, which lowers the compatibility cost of moving to pure ESM.

For most projects running on Node.js 20+ and using the [JavaScript API](/api/javascript-api), this change should have little impact. However, for tooling environments that still depend on older CommonJS assumptions, upgrade-related adjustments may be needed.

For example, when upgrading to 2.0, you will usually upgrade the related packages to 2.x together:

```json title="package.json"
{
  "devDependencies": {
    "@rspack/core": "^2.0.0",
    "@rspack/cli": "^2.0.0",
    "@rspack/dev-server": "^2.0.0",
    "@rspack/plugin-react-refresh": "^2.0.0"
  }
}
```

### Better ESM

At the bundled output level, Rspack 2.0 further consolidates the ESM work introduced in the 1.x series. Previously, Rspack used the experimental [`EsmLibraryPlugin`](/guide/migration/rspack_1.x#removed-esmlibraryplugin) to address issues such as excessive runtime code in ESM output and limited support for static analysis after code splitting. In 2.0, that capability is further consolidated into more natural configuration interfaces such as [`modern-module`](/config/output#outputlibrarytype) and [`output.module`](/config/output#outputmodule).

This means the generated ESM output is cleaner, easier to analyze statically, and more suitable for library builds, code splitting, and downstream tree shaking.

For example, in a library build you can configure:

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
    library: {
      type: 'modern-module',
    },
  },
};
```

### Import meta properties

At the source code level, Rspack 2.0 further improves how ESM semantics are handled. In addition to existing support for [`import.meta.url`](/api/runtime-api/module-variables#importmetaurl), [`import.meta.webpackHot`](/api/runtime-api/module-variables#importmetawebpackhot), and [`import.meta.webpackContext`](/api/runtime-api/module-variables#importmetawebpackcontext), Rspack now adjusts how unknown `import.meta` properties are handled: they are no longer decided too early at compile time and can instead be preserved for runtime evaluation through [`module.parser.javascript.importMeta`](/config/module#moduleparserjavascriptimportmeta).

This makes it easier to support `import.meta.*` patterns introduced by frameworks, tools, and runtimes.

For example, if you want unknown `import.meta` properties to be preserved for runtime evaluation, you can configure:

```js title="rspack.config.mjs"
export default {
  output: {
    module: true,
  },
  module: {
    parser: {
      javascript: {
        importMeta: 'preserve-unknown',
      },
    },
  },
};
```

Rspack 2.0 also adds support for [`import.meta.main`](/api/runtime-api/module-variables#importmetamain), [`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename), [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname), and related properties.

#### import.meta path APIs

Rspack 2.0 also expands support for ESM metadata APIs such as [`import.meta.main`](/api/runtime-api/module-variables#importmetamain), [`import.meta.filename`](/api/runtime-api/module-variables#importmetafilename), [`import.meta.dirname`](/api/runtime-api/module-variables#importmetadirname), and [`import.meta.resolve`](/api/runtime-api/module-variables#importmetaresolve). Together, these APIs make it more natural to work with module identity, filesystem location, and module resolution in ESM code.

For `import.meta.resolve`, you need to enable `module.parser.javascript.importMetaResolve` explicitly:

```js title="rspack.config.mjs"
export default {
  module: {
    parser: {
      javascript: {
        importMetaResolve: true,
      },
    },
  },
};
```

```js title="app.mjs"
console.log(import.meta.filename);
console.log(import.meta.dirname);
const asset = import.meta.resolve('./img.png');
```

#### import.meta.webpackContext include and exclude

Rspack 2.0 also extends `import.meta.webpackContext`. You can now use `include` and `exclude` to narrow the match range more precisely, which is useful when importing many files but only wanting a subset of them in the final context.

```js title="app.js"
const context = import.meta.webpackContext('./components', {
  recursive: true,
  regExp: /\.tsx?$/,
  include: /\.client\./,
  exclude: /\.test\./,
});
```

### Rspack dev server 2.0

[`@rspack/dev-server`](/guide/migration/rspack_1.x#development-server) has been substantially reorganized in 2.0. It is no longer a default dependency of `@rspack/cli`, and is now an optional installation. Internally, it also adopts `@rspack/dev-middleware` and replaces the heavier Express v4-based setup with `connect-next`.

As a result, the dev server is lighter and better suited for integration by upper-layer frameworks. For standard usage, see the [Rspack Dev Server guide](/guide/features/dev-server).

If you use `rspack dev` or `rspack serve`, you now need to install `@rspack/dev-server` explicitly:

```bash
pnpm add -D @rspack/dev-server
```

### Module resolution improvements

Rspack 2.0 also improves module resolution in a few practical ways.

#### `#/` subpath alias imports

Rspack 2.0 supports `#/`-style subpath alias imports, so you can reuse the `imports` field in `package.json` to organize intra-package path mapping instead of maintaining a separate alias configuration.

```json title="package.json"
{
  "imports": {
    "#/*": "./src/*"
  }
}
```

```js title="src/index.js"
import main from '#/main.ts';
```

#### `resolve.pnpManifest`

For Yarn PnP scenarios, Rspack 2.0 also adds `resolve.pnpManifest`, allowing you to explicitly specify the path to the PnP manifest file. This gives more precise control over resolver behavior in more complex package-management and monorepo environments.

```js title="rspack.config.mjs"
import path from 'node:path';

export default {
  resolve: {
    pnp: true,
    pnpManifest: path.resolve(import.meta.dirname, '.pnp.cjs'),
  },
};
```

### Portable cache

When Rspack 1.0 was released, portable cache was still a next step in the cache roadmap. In 2.0, Rspack now provides an official [`cache.portable`](/config/cache#cacheportable) option. The main goal is to make cache entries not only persistent, but also reusable across different machines, paths, and environments, making them more suitable for team workflows, remote caching, and future distributed build scenarios.

It is still disabled by default, but it now has an official configuration entry and usage path.

For example:

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    portable: true,
  },
};
```

Rspack 2.0 also provides [`cache.readonly`](/config/cache#cachereadonly). When enabled, cache entries are read from disk but not written back, which is especially useful in CI where you want to reuse prewarmed cache data without mutating it.

```js title="rspack.config.mjs"
export default {
  cache: {
    type: 'persistent',
    readonly: Boolean(process.env.CI),
  },
};
```

### Automatic targets

Rspack 2.0 further reduces duplicated target configuration. The target-related options of [`builtin:swc-loader`](/guide/features/builtin-swc-loader), [`builtin:lightningcss-loader`](/guide/features/builtin-lightningcss-loader), [`SwcJsMinimizerRspackPlugin`](/plugins/rspack/swc-js-minimizer-rspack-plugin), and [`LightningCssMinimizerRspackPlugin`](/plugins/rspack/lightning-css-minimizer-rspack-plugin) now inherit from the top-level [`target`](/config/target) configuration by default, instead of requiring developers to keep those settings in sync manually.

This reduces duplicated configuration and helps avoid mismatches between transform and minification targets.

For example, loader and plugin targets now inherit from the top-level `target` by default. If you need to override one loader, you can still configure it explicitly:

```js title="rspack.config.mjs"
export default {
  target: 'node',
  module: {
    rules: [
      {
        test: /\.js$/,
        loader: 'builtin:swc-loader',
        options: {
          env: {
            targets: 'chrome >= 87',
          },
        },
      },
    ],
  },
};
```

### Better peer dependencies

Rspack 2.0 also further refines dependency boundaries. For example, `@rspack/dev-server` is now an optional peer dependency of `@rspack/cli`, and `@module-federation/runtime-tools` has also been changed from a direct dependency to an optional peer dependency, with a wider compatible version range.

These adjustments make dependency ownership clearer and give upper-layer frameworks and tools more flexibility during integration.

For example, if you use `ModuleFederationPlugin`, you now need to install `@module-federation/runtime-tools` explicitly:

```bash
pnpm add @module-federation/runtime-tools
```

### Import defer

Rspack first added support for `import defer` in 1.6, and in 2.0 that support continues to expand. In addition to the original `import defer * as foo from './foo'`, Rspack later added support for `import.defer()`, allowing the "import now, execute later" model to cover more real-world cases.

This gives developers finer control over execution timing and side-effect timing.

For example:

```js title="rspack.config.mjs"
export default {
  experiments: {
    deferImport: true,
  },
};
```

```js title="app.js"
import defer * as foo from './foo';

console.log(foo.bar);
```

### MF share treeshaking

In [Module Federation](/guide/features/module-federation), Rspack 2.0 continues improving shared dependency pruning. With MF share treeshaking, Rspack can trim `shared` dependencies at a finer export level, reducing unused parts of shared packages and further shrinking dependency size in micro-frontend scenarios.

It supports both `runtime-infer` and `server-calc` modes, targeting two different engineering strategies: runtime inference and deployment-side secondary calculation.

For example:

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [
    new rspack.container.ModuleFederationPlugin({
      shared: {
        'lodash-es': {
          singleton: true,
          treeShaking: {
            mode: 'runtime-infer',
            usedExports: ['debounce'],
          },
        },
      },
    }),
  ],
};
```

### Import bytes

Rspack 2.0 also supports importing static assets as bytes. This allows you to load a resource as a `Uint8Array` and process it at runtime with APIs such as `TextDecoder`.

```js title="app.js"
import fileBytes from './file.bin' with { type: 'bytes' };

const decoder = new TextDecoder('utf-8');
const text = decoder.decode(fileBytes);
```

### builtin:swc-loader detectSyntax

Rspack 2.0 also adds [`detectSyntax`](/guide/features/builtin-swc-loader#detectsyntax) for `builtin:swc-loader`. When a single rule handles `.js`, `.jsx`, `.ts`, and `.tsx` together, `detectSyntax: 'auto'` can infer the parser configuration from the file extension and reduce repeated configuration.

```js title="rspack.config.mjs"
export default {
  module: {
    rules: [
      {
        test: /\.(?:js|mjs|jsx|ts|tsx)$/,
        use: {
          loader: 'builtin:swc-loader',
          options: {
            detectSyntax: 'auto',
          },
        },
      },
    ],
  },
};
```

### Better tree shaking

Rspack 2.0 further improves [tree shaking](/guide/optimization/tree-shaking) on top of the 1.x work. In 1.6, Rspack already improved tree shaking for dynamic imports. In 2.0, it also adds [`experiments.pureFunctions`](/guide/migration/rspack_1.x#new-experimentspurefunctions-for-function-level-tree-shaking), allowing finer-grained analysis of cross-module pure function calls through `/*#__NO_SIDE_EFFECTS__*/` annotations or [`module.parser.javascript.pureFunctions`](/config/module#moduleparserjavascriptpurefunctions).

Combined with the constant inlining, enum inlining, and barrel optimization work already introduced in 1.x, this extends dead-code elimination from module-level optimization toward function-level optimization.

For example:

```js title="rspack.config.mjs"
export default {
  experiments: {
    pureFunctions: true,
  },
  module: {
    rules: [
      {
        test: /node_modules\/some-library\/index.js/,
        parser: {
          pureFunctions: ['isString', 'default'],
        },
      },
    ],
  },
};
```

### Hashed module IDs plugin

Rspack 2.0 also adds `HashedModuleIdsPlugin`. If you want more explicit control over module ID generation instead of relying only on default behavior or the string values of `optimization.moduleIds`, you can use this plugin directly.

```js title="rspack.config.mjs"
import { rspack } from '@rspack/core';

export default {
  plugins: [new rspack.ids.HashedModuleIdsPlugin()],
};
```

### splitChunks enforceSizeThreshold

Rspack 2.0 also adds `splitChunks.enforceSizeThreshold`. It is useful when you want to set a hard size threshold for chunk splitting; once the module group size exceeds that threshold, some `maxRequests` limits are ignored so splitting can proceed more aggressively.

```js title="rspack.config.mjs"
export default {
  optimization: {
    splitChunks: {
      enforceSizeThreshold: 50000,
    },
  },
};
```

### Built-in React Server Components

Rspack 2.0 provides built-in [React Server Components](/guide/tech/rsc) support. Instead of relying entirely on upper-layer frameworks to assemble everything, Rspack now provides core-level plugins and compilation support for RSC workflows. Through [`rsc.createPlugins()`](/guide/tech/rsc#configure-plugins), Rspack can coordinate Server Compiler and Client Compiler pipelines, collect `"use client"`, `"use server"`, and `"use server-entry"` directives, generate the manifests required by the RSC runtime, and expose update signals through [`onServerComponentChanges`](/guide/tech/rsc#development-server) when server components change.

At the same time, `builtin:swc-loader` has been enhanced to recognize and transform these RSC directives, allowing React 19-era RSC, Server Actions, client references, and server references to work together inside one build system.

For example:

```js title="rspack.config.mjs"
import { experiments } from '@rspack/core';

const { createPlugins } = experiments.rsc;
const { ServerPlugin, ClientPlugin } = createPlugins();

export default [
  {
    target: 'web',
    plugins: [new ClientPlugin()],
  },
  {
    target: 'node',
    plugins: [new ServerPlugin()],
  },
];
```

## FAQ

### What should I check first when upgrading to Rspack 2.0?

If you use Rspack through the [JavaScript API](/api/javascript-api) or through an upper-layer tool integration, the most important things to check first are the module-system and dependency-boundary changes. In particular, `@rspack/core`, `@rspack/cli`, `@rspack/dev-server`, and `@rspack/plugin-react-refresh` are now [pure ESM packages](/guide/migration/rspack_1.x#pure-esm-packages), and [`@rspack/dev-server`](/guide/migration/rspack_1.x#development-server) is no longer installed by default with `@rspack/cli`. If your environment still assumes CommonJS usage patterns or automatic dev-server installation, plan to update those assumptions during the upgrade.

### Is portable cache enabled by default?

No. [`cache.portable`](/config/cache#cacheportable) now has an official configuration entry in Rspack 2.0, but it remains disabled by default. You can enable it when you want cache reuse across machines, paths, or environments.

### Is Rspack 2.0 mainly about performance?

No. Performance is still one of Rspack's core strengths, but 2.0 is not only about performance. It also focuses on compatibility, output quality, module capabilities, and continued improvement to the overall engineering experience.

### Does Rspack stop being compatible with webpack after 2.0?

No. Rspack will continue to maintain a limited level of webpack compatibility and will still try to keep migration and adoption practical for existing projects. The difference is that, after 2.0, compatibility is no longer the only goal. When we find approaches that are better in areas such as build performance, bundle size, community conventions, or Agent support, and those approaches require breaking some historical compatibility, we may choose to make those changes.

Even in those cases, we will still try to provide detailed transition guidance, upgrade notes, and migration paths so users can understand the change and move forward with a predictable cost.

### After 2.0 is released, will the 1.x line stop being maintained entirely?

No. After 2.0, Rspack's main direction will gradually shift to 2.x, but that does not mean the 1.x line will immediately stop receiving all maintenance. For projects still using 1.x, we will provide limited maintenance as needed in practice, such as critical fixes or migration-period support.

That said, in the longer term, new capabilities and major improvements will primarily move forward on 2.x. If your project is still on 1.x, we recommend planning the upgrade sooner rather than later.
